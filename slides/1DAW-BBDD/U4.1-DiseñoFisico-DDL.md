# 1) Introducción: Del diseño lógico al físico

## El flujo de trabajo en Bases de Datos

```
Modelo E/R → Modelo Relacional → SQL DDL (Oracle)
```

| Fase | Descripción | Ejemplo |
|------|-------------|---------|
| **Diseño Conceptual** | Describe el negocio (entidades y relaciones) | Entidad `Usuario` con atributos |
| **Diseño Lógico** | Transforma a tablas, claves y relaciones | Tabla `usuarios(id, email, nombre)` |
| **Diseño Físico** | Implementa en un SGBD concreto | `CREATE TABLE usuarios (...)` en Oracle |

> **Idea clave**: No solo memorizar comandos, sino entender **por qué** se usan.

---

# ¿Qué es DDL y diferencia con DML y DCL?

| Lenguaje | Significado | Función | Ejemplos |
|----------|-------------|---------|----------|
| **DDL** | Data Definition Language | Crea o modifica **estructura** | `CREATE`, `ALTER`, `DROP` |
| **DML** | Data Manipulation Language | Inserta, actualiza, consulta **datos** | `INSERT`, `UPDATE`, `SELECT` |
| **DCL** | Data Control Language | Gestiona **permisos** | `GRANT`, `REVOKE` |


<div class="two-cols">
<div>

```sql
-- DDL: crea estructura
CREATE TABLE categorias (
    id_categoria NUMBER PRIMARY KEY,
    nombre VARCHAR2(60) NOT NULL
);

```
</div>
<div>

```sql 
-- DML: inserta datos
INSERT INTO categorias VALUES (1, 'Informática');

-- DCL: da permisos
GRANT SELECT ON categorias TO alumno_lectura;
```
</div>
</div>

# 2) Esquemas y Usuarios en Oracle

### Concepto fundamental: ¿Qué es un Esquema?

En Oracle, **usuario** y **esquema** están íntimamente relacionados:

| Concepto | Definición | Analogía |
|----------|------------|----------|
| **Usuario** | Cuenta para conectarse a la BD | Tu login en el sistema |
| **Esquema** | Contenedor de objetos (tablas, vistas, índices...) | Tu carpeta de trabajo |

> **Regla de Oracle**: Cuando creas un usuario, automáticamente se crea su esquema homónimo. Son prácticamente lo mismo.

```
Usuario "VENTAS" → Esquema "VENTAS" (contiene sus tablas, vistas, etc.)
```

---

# ¿Por qué es importante el Esquema?

<div class="two-cols">
<div>

### Organización lógica:
- Cada proyecto/aplicación tiene su propio esquema
- Los objetos de diferentes esquemas no se mezclan
- Puedes tener tablas con el mismo nombre en esquemas distintos

### Seguridad:
- Los permisos se otorgan por esquema
- Un usuario no ve (por defecto) los objetos de otros esquemas

</div>
<div>

### Ejemplo práctico:
```sql
-- Esquema VENTAS: tabla clientes
VENTAS.clientes

-- Esquema RRHH: tabla clientes (misma BD, diferente esquema)
RRHH.clientes
```

</div>
</div>

---

# Crear Usuario y Esquema en Oracle

<div class="two-cols">
<div>

### Paso 1: Conectar como administrador (SYS o SYSTEM)
```sql
sqlplus / as sysdba
-- o
sqlplus system/tu_password
```

### Paso 2: Crear el usuario (que será el esquema)
```sql
CREATE USER ventas IDENTIFIED BY ventas123
DEFAULT TABLESPACE users
QUOTA UNLIMITED ON users;
```

</div>
<div>

### Paso 3: Dar permisos necesarios (DCL)
```sql
-- Permiso básico para conectarse
GRANT CREATE SESSION TO ventas;

-- Permisos para crear objetos DDL
GRANT CREATE TABLE, CREATE VIEW, 
      CREATE SEQUENCE, CREATE SYNONYM, 
      CREATE TRIGGER, CREATE INDEX,
      CREATE PROCEDURE TO ventas;
```
</div>
</div>

---

# Permisos de Sistema vs Permisos de Objeto

<div class="two-cols">
<div>

### Permisos de Sistema (GRANT... TO usuario):
| Permiso | Permite crear... |
|---------|------------------|
| `CREATE SESSION` | Conectarse a la BD |
| `CREATE TABLE` | Tablas en su esquema |
| `CREATE VIEW` | Vistas |
| `CREATE SEQUENCE` | Secuencias |
| `CREATE TRIGGER` | Disparadores |
| `CREATE ANY TABLE` | Tablas en **cualquier** esquema (¡admin!) |

</div>
<div>

### Permisos de Objeto (sobre objetos existentes):
```sql
-- Dar acceso a otros usuarios
GRANT SELECT, INSERT ON ventas.clientes TO app_web;
GRANT ALL ON ventas.pedidos TO admin_sistema;
```
</div>
</div>

# 3) SQL*Plus: Herramienta Profesional de Oracle

## ¿Qué es SQL*Plus?

SQL*Plus es la **interfaz de línea de comandos** original de Oracle. Permite:

- Ejecutar sentencias SQL y bloques PL/SQL
- Formatear resultados en informes profesionales
- Automatizar tareas mediante scripts (.sql)
- Administrar la base de datos

---

# Conexión a SQL*Plus:

```bash
# Formato básico
sqlplus usuario/password

# Conexión a BD específica
sqlplus usuario/password@nombre_bd

# Como administrador (SYSDBA)
sqlplus / as sysdba

# Modo silencioso (para scripts)
sqlplus -s usuario/password @script.sql
```

---

# Comandos SQL*Plus Esenciales

<div class="two-cols">
<div>

### Navegación y Sesión
| Comando | Descripción |
|---------|-------------|
| `CONNECT usuario/password` | Cambiar de usuario sin salir |
| `DISCONNECT` | Desconectar pero mantener SQL*Plus abierto |
| `EXIT` o `QUIT` | Salir de SQL*Plus |
| `SHOW USER` | Muestra usuario conectado |
| `CLEAR SCREEN` | Limpia la pantalla |

</div>
<div>

### Ejecución de Scripts
| Comando | Descripción |
|---------|-------------|
| `START archivo.sql` o `@archivo.sql` | Ejecuta script |
| `@@archivo.sql` | Ejecuta script desde ruta relativa al padre |
| `ED archivo.sql` | Abre editor externo (notepad/vi) |
| `GET archivo.sql` | Carga script en buffer sin ejecutar |
| `/` | Re-ejecuta último comando del buffer |

</div>
</div>

# Edición de Comandos en SQL*Plus

## Buffer de edición:
```sql
-- Ver último comando ejecutado
LIST
-- o
L

-- Editar línea específica (ej: línea 2)
2
-- Cambia el texto de esa línea
CHANGE /nombre/nom_cliente/

-- Añadir línea después de la 2
INPUT WHERE id > 100;

-- Borrar línea 3
DEL 3

-- Ejecutar buffer modificado
/
```
---

## Variables de sustitución (interactivo):
```sql
-- &variable: pide valor al usuario
SELECT * FROM empleados WHERE departamento = &num_dept;

-- &&variable: define y reutiliza
DEFINE num_dept = 10
SELECT * FROM empleados WHERE departamento = &num_dept;
UNDEFINE num_dept
```

# 4) Tipos de Objetos de una Base de Datos

| Objeto | Descripción | Uso principal |
|--------|-------------|---------------|
| **Tabla** | Unidad básica de almacenamiento, formada por filas y columnas | Guardar datos de negocio |
| **Vista** | Representación lógica de un subconjunto de datos (tabla virtual) | Simplificar consultas complejas |
| **Secuencia** | Estructura que genera valores automáticos (IDs consecutivos) | Crear identificadores únicos |
| **Índice** | Estructura auxiliar que acelera búsquedas | Mejorar rendimiento de consultas |
| **Sinónimo** | Alias o nombre alternativo para otros objetos | Simplificar nombres largos |
| **Trigger** | Bloque de código que se ejecuta automáticamente | Validaciones complejas, auditoría |

> **Nota**: En Oracle, el diccionario de datos almacena información sobre todos estos objetos en vistas como `USER_TABLES`, `USER_VIEWS`, `USER_INDEXES`, etc.

---

# Reglas de Nomenclatura en Oracle

<div class="two-cols">
<div>

## Convenciones obligatorias:
- Deben **empezar con una letra** y no superar los **30 caracteres**
- Solo caracteres: `A-Z`, `a-z`, `0-9`, `_`, `$`, `#`
- No duplicar nombres para el mismo usuario (concepto de **esquema**)
- **No usar palabras reservadas** (`USER`, `ORDER`, `SELECT`, `TABLE`...)

</div>
<div>

## Buenas prácticas recomendadas:
- Usar nombres **descriptivos** y consistentes (`snake_case`)
- Mantener coherencia entre claves relacionadas (`id_usuario` en ambas tablas)
- Nombrar restricciones con prefijos: `pk_`, `fk_`, `ck_`, `uq_`
- **No es sensible a mayúsculas/minúsculas**, pero por convención usamos mayúsculas para SQL

</div>
</div>

---

# Reglas de Nomenclatura en Oracle

| Correcto | Incorrecto | Motivo |
|----------|------------|--------|
| `usuarios` | `tabla1` | No describe el contenido |
| `fecha_alta` | `f1` | Poco legible |
| `id_producto` | `idprod` | Inconsistencia |
| `fk_pedidos_usuarios` | `fk1` | Difícil de mantener |

---

# Tipos de Datos en Oracle

###  Alfanuméricos:
| Tipo | Descripción | Cuándo usar |
|------|-------------|-------------|
| `VARCHAR2(n)` | Longitud **variable**. Ajusta al tamaño real del valor | Campos con longitud controlada (nombres, emails) |
| `CHAR(n)` | Longitud **fija**. Rellena con espacios en blanco | Códigos de tamaño fijo (DNI, matrículas) |
| `NCHAR` / `NVARCHAR2` | Usan caracteres **Unicode** | Texto en múltiples idiomas/alfabetos 

---

# Tipos de Datos en Oracle

### Numéricos y Fecha:
| Tipo | Descripción | Cuándo usar |
|------|-------------|-------------|
| `NUMBER(p, s)` | Números. `p`=dígitos totales, `s`=decimales | Precios, cantidades, IDs |
| `NUMBER(1)` | Alternativa para booleanos (0/1) | Campos sí/no, activo/inactivo |
| `DATE` | Fecha y hora (precisión de segundos) | Fechas de nacimiento, altas |
| `TIMESTAMP` | Fecha y hora con fracciones de segundo | Auditoría, logs precisos |
| `TIMESTAMP WITH TIME ZONE` | Incluye zona horaria | Sistemas internacionales |

> **Nota**: En Oracle NO usamos `BOOLEAN`. Usamos `NUMBER(1)` o `CHAR(1)`.

---

# VARCHAR2 vs CLOB, DATE vs TIMESTAMP

```sql
-- VARCHAR2: limita longitud máxima por negocio
CREATE TABLE productos (
    titulo VARCHAR2(150),   -- Máximo 150 caracteres
    descripcion CLOB        -- Texto muy largo (sin límite práctico)
);

-- DATE: precisión de segundos
-- TIMESTAMP: precisión de milisegundos
CREATE TABLE auditoria (
    fecha_evento DATE,           -- '08/04/2026 10:30:45'
    instante_evento TIMESTAMP    -- '08/04/2026 10:30:45.123456'
);
```
---
# VARCHAR2 vs CLOB, DATE vs TIMESTAMP

| Decisión | Tipo recomendado |
|----------|------------------|
| `fecha_nacimiento` | `DATE` |
| Registrar login exacto | `TIMESTAMP` |
| Descripción de producto larga | `CLOB` |
| Email del usuario | `VARCHAR2(120)` |

# 5) Manipulación de Tablas

## Creación básica:
```sql
CREATE TABLE nombre_tabla (

    nombre_columna2 tipo_dato [DEFAULT expresión],
```

### Ejercicio: Tabla Persona
Crear una tabla para almacenar datos personales: **dni, nombre, apellidos, fecha nacimiento, número de hermanos y estado civil** (s: soltero, c: casado, d: divorciado, v: viudo)

---

# Solución: Tabla Persona

```sql
CREATE TABLE persona (
    dni CHAR(9),
    nombre VARCHAR2(50),
    apellidos VARCHAR2(50),
    fecha_nac DATE,
    numero_hermanos NUMBER(2),
    estado_civil CHAR(1)
);
```

**Verificar estructura:**
```sql
DESCRIBE persona;
-- o
DESC persona;
```

---

# Valores por Defecto (DEFAULT)

`DEFAULT` asigna un valor automático cuando no se especifica en el `INSERT`.

**Ventajas:**
- Menos errores manuales
- Comportamiento consistente
- Código de aplicación más simple

###  Ejercicio: Añadir DEFAULT a Persona
Modificar el ejercicio anterior para que:
- `fecha_alta` sea por defecto la fecha actual del sistema
- `estado_civil` sea por defecto 'S' (soltero)

---

# Solución con DEFAULT

```sql
CREATE TABLE persona (
    dni CHAR(9),
    nombre VARCHAR2(50),
    apellidos VARCHAR2(50),
    fecha_alta DATE DEFAULT SYSDATE,  -- Fecha actual del sistema
    numero_hermanos NUMBER(2),
    estado_civil CHAR(1) DEFAULT 'S'   -- 'S' = Soltero
);
```

> **Nota**: `SYSDATE` es la función de Oracle para la fecha/hora actual. También existe `CURRENT_TIMESTAMP`.

---

# Comentarios sobre Tablas y Columnas

Los comentarios **documentan el esquema** dentro de la BD, facilitando el mantenimiento.

```sql
-- Comentario en tabla
COMMENT ON TABLE persona IS 
    'Datos referentes a los componentes de una asociación';

-- Comentario en columna
COMMENT ON COLUMN persona.estado_civil IS 
    'Valores válidos: S=Soltero, C=Casado, D=Divorciado, V=Viudo';
```

**Consultar comentarios:**

```sql
-- Tablas: ALL_TAB_COMMENTS, USER_TAB_COMMENTS
-- Columnas: ALL_COL_COMMENTS, USER_COL_COMMENTS
SELECT * FROM user_tab_comments WHERE table_name = 'PERSONA';
```

<div>

# Otras Operaciones con Tablas

### Renombrar tabla:
```sql
RENAME persona TO socios;
```

### Vaciar tabla (elimina filas, mantiene estructura):
```sql
-- TRUNCATE: rápido, no genera redo log, no se puede deshacer
TRUNCATE TABLE persona;
```
---

# Otras Operaciones con Tablas

### Eliminar tabla completamente:
```sql
-- DROP: elimina estructura y datos
DROP TABLE persona;

-- Con CASCADE CONSTRAINTS: elimina también restricciones dependientes
DROP TABLE persona CASCADE CONSTRAINTS;
```

| Comando | ¿Elimina filas? | ¿Elimina estructura? | ¿Reversible? |
|---------|-----------------|----------------------|--------------|
| `DELETE` | Sí | No | Sí (con ROLLBACK) |
| `TRUNCATE` | Sí | No | **No** |
| `DROP` | Sí | Sí | **No** |

# 6) Integridad de Datos y Constraints

### ¿Qué es la Integridad de Datos?

La **integridad de datos** es la garantía de que los datos almacenados son:
- **Correctos**: cumplen reglas de negocio
- **Consistentes**: no hay contradicciones entre tablas
- **Válidos**: respetan formatos y restricciones definidas

#### Tipos de Integridad
| Tipo | Descripción | Implementación |
|------|-------------|----------------|
| **Entidad** | Cada fila es única e identificable | `PRIMARY KEY`, `UNIQUE` |
| **Referencial** | Las relaciones entre tablas son válidas | `FOREIGN KEY` |
| **Dominio** | Los valores cumplen reglas de negocio | `CHECK`, `NOT NULL`, tipos de dato |
| **Usuario** | Reglas específicas de negocio | `CHECK`, `TRIGGER` |

---

# Constraints: Mecanismos de Integridad

<div class="two-cols">
<div>

Un **CONSTRAINT** (restricción) es una **regla que Oracle obliga a cumplir**:

- Previene datos no válidos en las tablas
- Protege la integridad de datos automáticamente
- Reduce errores en la aplicación (la BD valida por sí misma)

###  Funciones:
- Prevenir el borrado de una tabla si existen dependencias
- Hacer cumplir reglas al insertar, modificar o borrar datos
- Garantizar que las relaciones entre tablas sean válidas

</div>
<div>

### Tipos de Constraints:
| Restricción | Función | Integridad |
|-------------|---------|------------|
| `NOT NULL` | Obliga a informar valor | Dominio |
| `UNIQUE` | Evita duplicados | Entidad |
| `PRIMARY KEY` | Identifica cada fila (única + no nula) | Entidad |
| `FOREIGN KEY` | Asegura relación válida entre tablas | Referencial |
| `CHECK` | Valida condiciones personalizadas | Dominio |

</div>
</div>

---

# Relación entre Constraints e Integridad

<div class="two-cols">
<div>

### Integridad de Entidad
**Garantiza que cada fila sea única y se pueda identificar**

**¿Por qué importa?** Sin integridad de entidad, no podrías:
- Referenciar filas desde otras tablas (FK)
- Actualizar o borrar filas específicas
- Evitar duplicados lógicos

</div>
<div>

```sql
-- PRIMARY KEY: identificador único de la tabla
CREATE TABLE departamentos (
    id_depto NUMBER CONSTRAINT dept_id_pk PRIMARY KEY,
    nombre VARCHAR2(50)
);

-- UNIQUE: valores alternativos únicos (emails, DNI...)
CREATE TABLE empleados (
    id_emp NUMBER PRIMARY KEY,
    email VARCHAR2(100) CONSTRAINT emp_email_uq UNIQUE,
    dni VARCHAR2(9) CONSTRAINT emp_dni_uq UNIQUE
);
```
</div>
</div>

---

# Integridad Referencial

<div class="two-cols">
<div>

**Garantiza que las relaciones entre tablas sean válidas**

**Reglas de integridad referencial:**
1. No puedes insertar un empleado con `id_depto` inexistente
2. No puedes borrar un departamento que tenga empleados (salvo CASCADE)
3. No puedes modificar el ID de departamento si tiene empleados

</div>
<div>

```sql
-- Tabla padre (debe existir primero)
CREATE TABLE departamentos (
    id_depto NUMBER PRIMARY KEY,
    nombre VARCHAR2(50)
);

-- Tabla hija (referencia a padre)
CREATE TABLE empleados (
    id_emp NUMBER PRIMARY KEY,
    nombre VARCHAR2(50),
    id_depto NUMBER,
    CONSTRAINT emp_depto_fk FOREIGN KEY (id_depto)
        REFERENCES departamentos(id_depto)
        ON DELETE SET NULL  -- Si borran depto, empleado queda sin depto
);
```
</div>
</div>

---

# Integridad de Dominio

**Garantiza que los valores estén dentro de un conjunto permitido**

**¿Por qué importa?** Evita datos absurdos o inconsistentes (precios negativos, stock -5, categorías inventadas).

```sql
CREATE TABLE productos (
    id_producto NUMBER PRIMARY KEY,
    nombre VARCHAR2(100) NOT NULL,  -- Dominio: debe tener nombre
    precio NUMBER(10,2) 
        CONSTRAINT prod_precio_ck CHECK (precio > 0),  -- Dominio: precio positivo
    categoria VARCHAR2(20)
        CONSTRAINT prod_cat_ck CHECK (categoria IN ('A', 'B', 'C')),  -- Dominio: valores limitados
    stock NUMBER DEFAULT 0
        CONSTRAINT prod_stock_ck CHECK (stock >= 0),  -- Dominio: no negativo
    activo NUMBER(1) DEFAULT 1
        CONSTRAINT prod_activo_ck CHECK (activo IN (0,1))  -- Dominio: booleano
);
```

# 8) Tipos de Constraints en Detalle

## NOT NULL: Integridad de Dominio

**Solo a nivel de columna.** Obliga a que la columna tenga valor.

```sql
CREATE TABLE alumnos (
    dni CHAR(9),
    nombre VARCHAR2(50),
    apellidos VARCHAR2(50),
    telefono CHAR(9) NOT NULL  -- Obligatorio
);
```

**Ejercicio**: Crear tabla de alumnos de 1º DAM con DNI, nombre, apellidos y teléfono móvil obligatorio.

---

# UNIQUE: Integridad de Entidad

Evita valores duplicados. Permite múltiples por tabla.

**Diferencia con PRIMARY KEY:**
- `PRIMARY KEY`: única + no nula + una por tabla
- `UNIQUE`: única, permite NULL, varias por tabla

```sql
CREATE TABLE zapateria (
    id_zapato VARCHAR2(9),
    marca VARCHAR2(50),
    modelo VARCHAR2(50),
    color VARCHAR2(9),
    CONSTRAINT zapateria_marca_uq UNIQUE(marca)  -- Solo una marca
);
```

>**Ejercicio**: Zapatería con convenio: solo un modelo por marca.

---

# PRIMARY KEY: Identificación Única

Incluye `UNIQUE` + `NOT NULL`. Una por tabla.

```sql
CREATE TABLE zapateria (
    id_zapato VARCHAR2(9),
    marca VARCHAR2(50),
    modelo VARCHAR2(50),
    color VARCHAR2(9),
    CONSTRAINT zapateria_id_pk PRIMARY KEY (id_zapato)
);
```

---

# PRIMARY KEY: Identificación única

### Clave primaria simple vs compuesta
| Tipo | Cuándo usar | Ejemplo |
|------|-------------|---------|
| Simple | Una columna identifica naturalmente | `id_usuario` |
| Compuesta | La unicidad depende de combinación | `matriculas(id_alumno, id_modulo, curso)` |

**Error típico**: Usar PK compuesta cuando conviene un ID artificial + `UNIQUE`.

---

# FOREIGN KEY: Integridad Referencial

**Garantiza que el valor hijo exista en la tabla padre.**

<br>

```sql
CREATE TABLE motos (
    matricula VARCHAR2(9),
    marca VARCHAR2(50),
    modelo VARCHAR2(50),
    color VARCHAR2(9),
    dni_propietario CHAR(9),
    CONSTRAINT motos_mat_pk PRIMARY KEY (matricula),
    CONSTRAINT motos_dni_fk FOREIGN KEY (dni_propietario)
        REFERENCES propietarios(dni_propietarios)
        ON DELETE CASCADE
);
```

---

# FOREIGN KEY: Integridad Referencial

### Opciones ON DELETE

<br>

| Opción | Comportamiento | Uso |
|--------|----------------|-----|
| `ON DELETE CASCADE` | Borra hijos automáticamente | Cuando los hijos no tienen sentido sin padre |
| `ON DELETE SET NULL` | Pone FK a NULL | Conservar histórico, requiere columna nullable |
| Sin cláusula (por defecto) | Bloquea borrado si hay hijos | Protección máxima, control manual |

# Vistas: Tablas Virtuales

```sql
       u.nombre AS cliente,

JOIN usuarios u ON u.id_usuario = p.id_usuario;
```

# Verificación de Integridad

```sql
-- Insertar usuario válido
INSERT INTO usuarios (nombre, email) 
VALUES ('Ana Ruiz', 'ana@daw.test');

-- Esto FALLA: viola integridad referencial
INSERT INTO pedidos (id_usuario) VALUES (9999);
-- ERROR: integrity constraint violated - parent key not found

-- Esto FALLA: viola CHECK constraint
INSERT INTO productos (nombre, precio) VALUES ('Ratón', -10);
-- ERROR: check constraint violated
```

---

# Errores Frecuentes y Soluciones

| Error | Causa | Solución |
|-------|-------|----------|
| `ORA-00942: table or view does not exist` | Tabla no creada o no tienes permisos | Crear tabla primero o verificar permisos |
| `ORA-02291: integrity constraint violated - parent key not found` | FK apunta a valor inexistente | Insertar primero en tabla padre |
| `ORA-02292: integrity constraint violated - child record found` | Intentas borrar padre con hijos | Usar `ON DELETE CASCADE` o borrar hijos primero |

---

# Errores Frecuentes y Soluciones

| Error | Causa | Solución |
|-------|-------|----------|
| `ORA-00001: unique constraint violated` | Valor duplicado en UNIQUE/PK | Verificar datos antes de insertar |
| `ORA-02290: check constraint violated` | No cumple condición CHECK | Revisar reglas de negocio |
| `ORA-01950: no privileges on tablespace` | Usuario sin cuota en tablespace | `ALTER USER ... QUOTA ... ON ...` |
