# 📦 Parte 1: Imágenes Docker

## ¿Qué es una Imagen Docker?

Una **imagen** es una plantilla inmutable de solo lectura que contiene:
- Un sistema de archivos completo
- Parámetros de configuración
- Aplicaciones y dependencias

<div class="highlight">

**Imagen vs Contenedor**
- **Imagen** = Plantilla estática (como un DVD de instalación)
- **Contenedor** = Instancia en ejecución (como el programa instalado)

</div>

---

# 🏗️ Características de las Imágenes

| Característica | Descripción |
|---------------|-------------|
| **Solo lectura** | No se pueden modificar directamente |
| **Basadas en capas** | Construidas mediante capas superpuestas |
| **Reutilizables** | Múltiples contenedores usan la misma imagen |
| **Versionables** | Etiquetadas con diferentes versiones |

<div class="success">

**Ventaja clave:** Las capas compartidas se almacenan **una sola vez** en disco, ahorrando espacio.

</div>

---

# 🏷️ Nomenclatura de Imágenes

```
[usuario/]nombre:etiqueta
```

**Componentes:**
- `usuario` → Organización que creó la imagen (omitido en oficiales)
- `nombre` → Nombre descriptivo
- `etiqueta` → Versión o variante

**Ejemplos prácticos:**
```bash
ubuntu:22.04              # Oficial: Ubuntu 22.04
mysql:8.0                 # Oficial: MySQL 8.0
nginx:latest              # Última versión (¡cuidado!)
josedom24/app_web:v1      # Imagen de usuario
```

<div class="warning">

⚠️ **¡Atención con `latest`!** No significa "la más reciente automáticamente". Es solo una etiqueta por defecto que no se actualiza sola.

</div>

---

# 🔍 Gestión de Imágenes: Comandos Esenciales

### Ver imágenes locales
```bash
$ docker images
REPOSITORY    TAG       IMAGE ID       CREATED        SIZE
ubuntu        latest    f63181f19b2f   7 days ago     72.9MB
nginx         1.21      ad4c705f24d3   2 weeks ago    133MB
```

**Columnas importantes:**
- `REPOSITORY` → Nombre de la imagen
- `TAG` → Versión (latest, 22.04, 8.0...)
- `IMAGE ID` → Hash único (mismo ID = imagen idéntica)
- `SIZE` → Espacio en disco

### Descargar imágenes
```bash
$ docker pull ubuntu:22.04     # Versión específica
$ docker pull nginx            # Equivale a nginx:latest
```

---

# 🗑️ Eliminación y Búsqueda

### Eliminar imágenes
```bash
$ docker rmi ubuntu:22.04          # Por nombre
$ docker rmi f63181f19b2f          # Por IMAGE ID
$ docker rmi -f ubuntu:22.04       # Forzar eliminación
```

<div class="warning">

⚠️ **Restricción:** No puedes eliminar una imagen si existen contenedores (incluso detenidos) creados a partir de ella.

</div>

### Buscar en Docker Hub
```bash
$ docker search nginx
$ docker search --filter is-official=true nginx   # Solo oficiales
$ docker search --filter stars=100 nginx          # Mínimo 100 estrellas
```

### Limpieza
```bash
$ docker image prune        # Eliminar imágenes sin usar
$ docker image prune -a     # Eliminar todas las no usadas
$ docker system df          # Ver espacio usado por Docker
```

---

# 🥞 Sistema de Capas (Layers)

Las imágenes Docker están compuestas por **capas ordenadas** de solo lectura.

<div class="columns">
<div>

**Ventajas del sistema:**
- ✅ **Reutilización:** Capas comunes se comparten
- ✅ **Eficiencia:** Solo se descargan capas nuevas
- ✅ **Rollback:** Fácil volver a versiones anteriores

</div>
<div>

**Ejemplo visual:**
```
nginx:latest
├── Capa base Ubuntu    (72 MB)
├── Capa dependencias   (+15 MB)
├── Capa Nginx binario  (+45 MB)
└── Capa configuración  (+1 MB)
    ───────────────────────
    Total: ~133 MB
```

</div>
</div>

---

# 📝 Capas en Contenedores: Copy-on-Write

Cuando creas un contenedor, Docker añade una **capa de escritura** (thin read-write layer).

```
┌──────────────────────────┐
│  Capa de escritura       │ ← Tus cambios van aquí
│  (Read-Write Layer)      │   (archivos nuevos/modificados)
├──────────────────────────┤
│  Capa 3: Aplicación      │ ↑
├──────────────────────────┤ │ Solo lectura
│  Capa 2: Dependencias    │ │ (compartidas)
├──────────────────────────┤ │
│  Capa 1: Sistema Base    │ ↓
└──────────────────────────┘
```

<div class="highlight">

**Copy-on-Write:** Cuando modificas un archivo existente, Docker lo copia a la capa de escritura antes de modificarlo. El original permanece intacto.

</div>

---

# 📊 Tamaño de Contenedores

### Ver tamaños
```bash
$ docker ps -a -s
CONTAINER ID   IMAGE    SIZE
abc123         ubuntu   0B (virtual 72.9MB)
```

**Interpretación:**
- **0B** → Tamaño de la capa de escritura (sin cambios)
- **virtual 72.9MB** → Capa escritura + imagen base

### Experimento práctico
```bash
# Crear archivo de 100MB en contenedor
$ docker exec contenedor1 dd if=/dev/zero of=archivo bs=1M count=100

# Ver resultado
$ docker ps -a -s
CONTAINER ID   IMAGE    SIZE
abc123         ubuntu   100MB (virtual 172.9MB)
```

<div class="success">

💡 **Eficiencia:** 10 contenedores de Ubuntu = 72.9MB (compartidos) + cambios individuales. ¡Mucho más eficiente que 10 VMs!

</div>

---

# 🏷️ Etiquetas y Versiones

### Etiquetar imágenes locales
```bash
# Crear alias local
$ docker tag ubuntu:22.04 mi-ubuntu:dev

# Ver resultado
$ docker images
REPOSITORY    TAG       IMAGE ID
ubuntu        22.04     f63181f19b2f
mi-ubuntu     dev       f63181f19b2f   # ¡Mismo ID!
```

### Buenas prácticas
1. ✅ Usar etiquetas específicas en producción (`ubuntu:22.04`)
2. ❌ Evitar `latest` en entornos críticos
3. ✅ Mantener imágenes actualizadas por seguridad
4. ✅ Preferir imágenes oficiales

# 💾 Parte 2: Almacenamiento Persistente

## Volúmenes y Bind Mounts

---

# 🔄 El Problema: Contenedores Efímeros

<div class="warning">

**Problema:** Los contenedores son efímeros. Cuando se eliminan, **todos los datos se pierden**.

</div>

**Soluciones de Docker:**

| Tipo | Uso | Persistencia |
|------|-----|--------------|
| **Volúmenes Docker** | Datos persistentes | ✅ Sí |
| **Bind Mounts** | Desarrollo/compartir archivos | ✅ Sí |
| **tmpfs mounts** | Datos temporales en RAM | ❌ No |

---

# 💽 Volúmenes Docker (Recomendado)

Gestionados completamente por Docker. Independientes del sistema de archivos del host.

### Comandos básicos
```bash
# Crear volumen
$ docker volume create mi_volumen

# Listar volúmenes
$ docker volume ls

# Inspeccionar
$ docker volume inspect mi_volumen
# Muestra: Mountpoint, Driver, Scope...

# Eliminar
$ docker volume rm mi_volumen
$ docker volume prune    # Eliminar no utilizados
```

---

# 🔗 Usar Volúmenes en Contenedores

### Sintaxis básica
```bash
$ docker run -d --name app   -v mi_volumen:/datos   ubuntu sleep infinity
```

**Formato:** `-v volumen:ruta_en_contenedor`

### Ejemplo: MySQL con datos persistentes
```bash
# 1. Crear volumen
$ docker volume create mysql_data

# 2. Ejecutar MySQL
$ docker run -d   --name mysql_db   -e MYSQL_ROOT_PASSWORD=secreto   -v mysql_data:/var/lib/mysql   mysql:8.0

# 3. Eliminar contenedor (¡datos seguros!)
$ docker rm -f mysql_db

# 4. Nuevo contenedor, mismos datos
$ docker run -d   --name mysql_db_nuevo   -v mysql_data:/var/lib/mysql   mysql:8.0
```

---

# 📂 Bind Mounts

Vinculan un directorio específico del host con el contenedor.

### Ideal para desarrollo
```bash
# Montar directorio del host
$ docker run -d --name web   -v /home/usuario/web:/usr/share/nginx/html   nginx

# Sintaxis alternativa (más explícita)
$ docker run -d --name web   --mount type=bind,source=/home/usuario/web,target=/usr/share/nginx/html   nginx
```

<div class="columns">
<div>

**Ventajas:**
- Acceso directo a archivos
- Cambios en tiempo real
- Compartir configuraciones

</div>
<div>

**Desventajas:**
- Dependen del host
- Menos portables
- Riesgo de seguridad

</div>
</div>

---

# 📊 Comparativa: Volúmenes vs Bind Mounts

| Característica | Volúmenes | Bind Mounts |
|---------------|-----------|-------------|
| **Gestión** | Docker | Usuario |
| **Portabilidad** | ✅ Alta | ❌ Baja |
| **Rendimiento** | Nativo | Depende del SO |
| **Uso ideal** | Producción | Desarrollo |
| **Backup** | Fácil | Manual |
| **Compartir datos** | Entre contenedores | Host ↔ Contenedor |

<div class="success">

**Regla general:** Usa **volúmenes** para datos de aplicaciones y **bind mounts** para desarrollo local.

</div>

# 🌐 Parte 3: Redes en Docker

## Comunicación entre Contenedores

---

# 🕸️ Tipos de Redes en Docker

| Tipo | Descripción | Uso |
|------|-------------|-----|
| **bridge** | Red privada interna (por defecto) | Aislamiento en un host |
| **host** | Usa la red del host directamente | Alto rendimiento, sin aislamiento |
| **none** | Sin conectividad de red | Contenedores aislados |
| **overlay** | Para múltiples hosts (Swarm) | Clústeres Docker |
| **macvlan** | MAC address propia | Dispositivo físico en red |

```bash
# Ver redes disponibles
$ docker network ls
NETWORK ID     NAME      DRIVER    SCOPE
a1b2c3d4e5f6   bridge    bridge    local
1234567890ab   host      host      local
```

---

# 🌉 Red Bridge por Defecto

Todos los contenedores se conectan automáticamente a la red `bridge`.

**Características:**
- Aislamiento entre contenedores
- Comunicación por IP (no por nombre)
- Puerto mapping necesario para acceso externo

```bash
# Inspeccionar red bridge
$ docker network inspect bridge

# Ver contenedores conectados y sus IPs
```

<div class="warning">

⚠️ **Limitación:** En la red bridge por defecto, los contenedores **no pueden comunicarse por nombre de host**, solo por IP.

</div>

---

# 🏗️ Redes Definidas por el Usuario

**Ventajas clave:**
- ✅ Resolución DNS automática por nombre
- ✅ Mejor aislamiento y seguridad
- ✅ Conexión/desconexión dinámica
- ✅ Configuración personalizada

### Crear red personalizada
```bash
# Simple
$ docker network create mi_red

# Con configuración específica
$ docker network create   --driver bridge   --subnet 172.20.0.0/16   --gateway 172.20.0.1   mi_red_custom
```

---

# 🔌 Conectar Contenedores a Redes

### Crear contenedor en red específica
```bash
$ docker run -d --name app --network mi_red nginx
```

### Conectar contenedor existente
```bash
# Añadir a red adicional (¡puede estar en varias!)
$ docker network connect mi_red contenedor1

# Desconectar
$ docker network disconnect mi_red contenedor1
```

### Ver IPs de un contenedor
```bash
$ docker inspect app | grep IPAddress
"IPAddress": "172.20.0.2",   # mi_red
"IPAddress": "172.17.0.2",   # bridge default
```

<div class="highlight">

💡 Un contenedor puede estar en **múltiples redes simultáneamente**.

</div>

---

# 💬 Comunicación entre Contenedores

### En red personalizada (DNS automático)
```bash
# Crear red
$ docker network create app_network

# Base de datos
$ docker run -d   --name db   --network app_network   -e MYSQL_ROOT_PASSWORD=secreto   mysql:8.0

# Aplicación (se conecta usando el nombre 'db')
$ docker run -d   --name app   --network app_network   -e DB_HOST=db   mi_aplicacion
```

**Conexión:** `mysql://db:3306/midb`

Docker resuelve `db` automáticamente a la IP del contenedor.

---

# 🧪 Ejemplo Práctico: WordPress + MySQL

### Paso 1: Infraestructura
```bash
# Red privada
$ docker network create wordpress_net

# Volúmenes para persistencia
$ docker volume create wp_db
$ docker volume create wp_files
```

### Paso 2: MySQL
```bash
$ docker run -d   --name wordpress_db   --network wordpress_net   -v wp_db:/var/lib/mysql   -e MYSQL_ROOT_PASSWORD=secreto   -e MYSQL_DATABASE=wordpress   -e MYSQL_USER=wpuser   -e MYSQL_PASSWORD=wppass   mysql:8.0
```

---

# 🧪 WordPress + MySQL (continuación)

### Paso 3: WordPress
```bash
$ docker run -d   --name wordpress   --network wordpress_net   -p 8080:80   -v wp_files:/var/www/html   -e WORDPRESS_DB_HOST=wordpress_db   -e WORDPRESS_DB_NAME=wordpress   -e WORDPRESS_DB_USER=wpuser   -e WORDPRESS_DB_PASSWORD=wppass   wordpress
```

### Diagrama de conexión
```
Navegador → localhost:8080 → WordPress (172.20.0.3)
                                      ↓
                              DNS: wordpress_db
                                      ↓
                              MySQL (172.20.0.2)
```

<div class="success">

✅ Accede a: `http://localhost:8080`

</div>

---

# 🧪 Verificar Persistencia

```bash
# 1. Instalar WordPress y crear contenido
# 2. Detener y eliminar contenedores
$ docker stop wordpress wordpress_db
$ docker rm wordpress wordpress_db

# 3. Recrear con los mismos comandos
# ¡Los datos persisten en los volúmenes!
```

**¿Por qué funciona?**
- `wp_db` → Base de datos MySQL
- `wp_files` → Archivos de WordPress
- `wordpress_net` → Comunicación por nombre

<div class="highlight">

💡 **Backup:** Solo necesitas respaldar los volúmenes `wp_db` y `wp_files`.

</div>

---

# 🧹 Limpieza y Mantenimiento

### Comandos útiles
```bash
# Limpiar redes sin uso
$ docker network prune

# Limpiar volúmenes sin uso
$ docker volume prune

# Limpiar imágenes sin uso
$ docker image prune -a

# Limpieza general (¡CUIDADO!)
$ docker system prune -a --volumes
```

<div class="warning">

⚠️ `docker system prune -a --volumes` elimina TODO lo no utilizado. Úsalo con precaución en producción.

</div>
