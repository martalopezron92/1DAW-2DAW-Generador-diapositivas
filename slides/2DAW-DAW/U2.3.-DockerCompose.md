# 1. ¿Qué es Docker Compose?

---

## 1. ¿Qué es Docker Compose?

Docker Compose permite **definir y ejecutar aplicaciones multi-contenedor** mediante un único archivo YAML.

En lugar de esto:

```bash
$ docker network create mi-app_default
$ docker run -d --name db --network mi-app_default \
    -e MYSQL_ROOT_PASSWORD=root mysql:8.0
$ docker run -d --name app --network mi-app_default \
    -p 8080:8080 mi-spring-app:latest
```

Hacemos esto:

```bash
$ docker compose up -d
```

---

## 1.1. Ventajas de Docker Compose

| Ventaja | Descripción |
|---|---|
| **Simplicidad** | Un solo archivo de configuración |
| **Reproducibilidad** | Mismo entorno en cualquier máquina |
| **Gestión centralizada** | Iniciar/detener toda la app con un comando |
| **Versionado** | El archivo se controla con Git |
| **Escalabilidad** | Fácil escalar servicios |

> **Regla de oro:** Si tu aplicación tiene más de un contenedor, usa Docker Compose.

# 2. Instalación y verificación

---

## 2. Instalación de Docker Compose

Docker Compose **v2** viene integrado como **plugin de Docker**.  
Se instala automáticamente con Docker Desktop o Docker Engine reciente.

**Verificar instalación:**

```bash
$ docker compose version
Docker Compose version v2.20.2
```

**Diferencia de versiones:**

| Versión | Comando | Tipo |
|---|---|---|
| v1 (obsoleta) | `docker-compose` | Binario independiente |
| **v2 (actual)** | `docker compose` | Plugin integrado |

> En este curso usaremos siempre la sintaxis **v2**: `docker compose`

# 3. El archivo docker-compose.yml

---

## 3.1. Estructura básica

```yaml
version: '3.8'   # Opcional en v2 (informativo)

services:
  # Cada servicio es un contenedor
  web:
    image: nginx
    ports:
      - "80:80"

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: secreto

volumes:
  # Volúmenes para persistir datos

networks:
  # Redes personalizadas (opcional)
```

---

## 3.1. Secciones del archivo

**`services:`** — La sección más importante. Define cada contenedor:

```yaml
services:
  nombre_servicio:   # Tú eliges el nombre
    image: nginx     # Imagen a usar
    ports:
      - "8080:80"    # host:contenedor
```

**`volumes:`** — Persistencia de datos:

```yaml
volumes:
  mi_volumen:        # Docker lo gestiona automáticamente
```

**`networks:`** — Redes de comunicación:

```yaml
networks:
  mi_red:
    driver: bridge   # Tipo de red (por defecto: bridge)
```

---

## 3.1. Equivalencia con comandos Docker

Este `docker-compose.yml`:

```yaml
services:
  web:
    image: nginx
    ports:
      - "8080:80"
```

Es equivalente a:

```bash
$ docker network create mi-app_default
$ docker run -d \
    --name web \
    --network mi-app_default \
    -p 8080:80 \
    nginx
```

> Con Compose, defines una vez y ejecutas con `docker compose up`.

---

## 3.2. Ejemplo completo: WordPress

```yaml
version: '3.8'
services:
  db:
    image: mysql:8.0
    volumes:
      - db_data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wpuser
      MYSQL_PASSWORD: wppass
    networks:
      - wordpress_net
    restart: always

  wordpress:
    image: wordpress:latest
    depends_on:
      - db
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: db       # ← nombre del servicio
      WORDPRESS_DB_USER: wpuser
      WORDPRESS_DB_PASSWORD: wppass
    networks:
      - wordpress_net

volumes:
  db_data:
networks:
  wordpress_net:
```

---

## 3.2. DNS interno de Docker Compose

```
WordPress busca el host: "db"
         ↓
DNS interno de Compose resuelve:
         db  →  172.18.0.2
         ↓
WordPress conecta a: mysql://172.18.0.2:3306
         ↓
MySQL responde ✅
```

**¡Magia!** Los servicios se comunican usando su **nombre de servicio** como hostname. No necesitas conocer IPs.

---

## 3.2. Opciones de `restart`

| Valor | Comportamiento |
|---|---|
| `no` | No reiniciar (por defecto) |
| `always` | Siempre reiniciar, incluso tras `docker compose stop` |
| `on-failure` | Solo si el proceso falla (exit code ≠ 0) |
| `unless-stopped` | Siempre, excepto si se detuvo manualmente |

```yaml
services:
  db:
    restart: unless-stopped   # Recomendado para producción
```

---

## 3.2. `depends_on`: orden de inicio

```yaml
services:
  app:
    depends_on:
      - db        # db inicia antes que app
```

> ⚠️ `depends_on` garantiza el **orden de inicio**, no que el servicio esté listo para aceptar conexiones.

**Solución:** usar healthcheck (lo veremos en buenas prácticas):

```yaml
services:
  app:
    depends_on:
      db:
        condition: service_healthy  # Espera al healthcheck de db

  db:
    healthcheck:
      test: ["CMD", "mysqladmin", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```

# 4. Comandos principales

---

## 4.1. Iniciar la aplicación

```bash
# Iniciar en segundo plano (detached) — lo más habitual
$ docker compose up -d
[+] Running 4/4
 ✔ Network wordpress_default       Created
 ✔ Volume "wordpress_db_data"      Created
 ✔ Container wordpress-db-1        Started
 ✔ Container wordpress-wordpress-1 Started
```

**Opciones útiles de `up`:**

```bash
# Ver logs directamente (primer plano)
$ docker compose up

# Forzar recreación de contenedores
$ docker compose up -d --force-recreate

# Reconstruir imágenes antes de iniciar
$ docker compose up -d --build

# Iniciar solo un servicio
$ docker compose up -d db
```

---

## 4.1. Ver logs

```bash
$ docker compose logs
wordpress-db-1        | 2024-01-23 17:00:00 [Note] mysqld: ready for connections.
wordpress-wordpress-1 | WordPress not found... copying files from /usr/src/wordpress
```

**Opciones de logs:**

```bash
# Últimas 50 líneas
$ docker compose logs --tail 50

# Seguir en tiempo real (como tail -f)
$ docker compose logs -f

# Solo un servicio
$ docker compose logs wordpress

# Con timestamps
$ docker compose logs -t

# Combinar opciones
$ docker compose logs -f --tail 100 db
```

---

## 4.1. Ver estado y gestionar servicios

```bash
# Ver estado de todos los servicios
$ docker compose ps
NAME                    IMAGE         SERVICE    STATUS   PORTS
wordpress-db-1          mysql:8.0     db         Up       3306/tcp
wordpress-wordpress-1   wordpress     wordpress  Up       0.0.0.0:8080->80/tcp

# Detener SIN eliminar (datos persisten)
$ docker compose stop

# Iniciar servicios detenidos
$ docker compose start

# Reiniciar
$ docker compose restart

# Pausar / Reanudar (congela procesos en memoria)
$ docker compose pause
$ docker compose unpause
```

---

## 4.1. Eliminar la aplicación

```bash
# Elimina contenedores y red, PERO conserva volúmenes
$ docker compose down

# Elimina también los volúmenes ⚠️ PIERDES LOS DATOS ⚠️
$ docker compose down -v

# Elimina también las imágenes descargadas
$ docker compose down --rmi all
```

**Diferencia entre `stop` y `down`:**

| Comando | Contenedores | Redes | Volúmenes |
|---|---|---|---|
| `stop` | Detenidos | Conservadas | Conservados |
| `down` | Eliminados | Eliminadas | Conservados |
| `down -v` | Eliminados | Eliminadas | **Eliminados** |

---

## 4.1. Ejecutar comandos en servicios

```bash
# Abrir shell en un contenedor en ejecución
$ docker compose exec wordpress bash
root@abc123:/var/www/html#

# Con usuario específico
$ docker compose exec -u www-data wordpress bash

# Ejecutar comando sin shell
$ docker compose exec wordpress ls -la

# Conectar a MySQL
$ docker compose exec db mysql -u root -p

# Backup de base de datos
$ docker compose exec db mysqldump -u root -prootpass wordpress > backup.sql

# Restaurar backup
$ docker compose exec -T db mysql -u root -prootpass wordpress < backup.sql
```

---

## 4.1. `exec` vs `run`

```bash
# exec: ejecuta en contenedor YA en marcha
$ docker compose exec app bash

# run: crea contenedor TEMPORAL para una tarea
$ docker compose run --rm app java -jar app.jar --version
```

| Comando | Contenedor | Uso típico |
|---|---|---|
| `exec` | Existente y en marcha | Inspección, debugging |
| `run` | Nuevo y temporal | Tareas puntuales, migraciones |

---

## 4.1. Escalar servicios

```bash
$ docker compose up -d --scale web=3
[+] Running 4/4
 ✔ Container proyecto-web-1  Started
 ✔ Container proyecto-web-2  Started
 ✔ Container proyecto-web-3  Started
 ✔ Container proyecto-db-1   Started
```

> ⚠️ No puedes escalar servicios con puertos específicos (`8080:80`), porque un solo puerto del host no puede usarlo más de un contenedor.

```yaml
services:
  web:
    expose:
      - "80"   # Puerto interno, sin mapeo fijo al host
    # ports:
    #   - "8080:80"   # ❌ Imposible escalar con esto
```

---

## 4.2. Opciones avanzadas de `docker compose`

```bash
# Usar un archivo compose diferente
$ docker compose -f mi-compose.yml up -d

# Combinar archivos (base + entorno)
$ docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

# Especificar nombre de proyecto
$ docker compose -p cliente1 up -d
$ docker compose -p cliente2 up -d   # Dos instancias independientes

# Ver configuración resultante (muy útil para debug)
$ docker compose config

# Validar archivo sin ejecutar
$ docker compose config --quiet
```

# 5. Definición de servicios

---

## 5.1. `image` vs `build`

**Usar imagen existente** (Docker Hub u otro registro):

```yaml
services:
  web:
    image: nginx:1.25.3
```

**Construir desde Dockerfile** (tu propio código):

```yaml
services:
  app:
    build:
      context: ./app          # Directorio con el Dockerfile
      dockerfile: Dockerfile  # Nombre del fichero (opcional)
      args:
        JAVA_VERSION: 21      # Argumentos de build
        PROFILE: production
      target: production      # Stage de multi-stage build
```

---

## 5.1. Dockerfile Java/Spring Boot típico

```
proyecto/
├── docker-compose.yml
└── app/
    ├── Dockerfile
    ├── pom.xml
    └── src/
```

```dockerfile
# app/Dockerfile
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src/ ./src/
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

```yaml
services:
  app:
    build: ./app
    ports:
      - "8080:8080"
```

---

## 5.2. `container_name`, `command` y `ports`

**`container_name`:** nombre personalizado para el contenedor:

```yaml
services:
  web:
    container_name: mi_nginx_custom
    # ⚠️ No se puede escalar si se usa container_name
```

**`command`:** sobrescribe el CMD de la imagen:

```yaml
services:
  app:
    image: eclipse-temurin:21-jre
    command: ["java", "-Dspring.profiles.active=dev", "-jar", "app.jar"]
```

**`ports`:** mapeo de puertos:

```yaml
ports:
  - "8080:8080"              # host:contenedor
  - "127.0.0.1:9090:9090"   # Solo localhost
  - "3000-3005:3000-3005"   # Rango de puertos
```

---

## 5.3. `environment` y archivos `.env`

**Formato inline (mapa):**

```yaml
services:
  app:
    environment:
      SPRING_PROFILES_ACTIVE: production
      DB_HOST: db
      DB_PORT: "5432"
      DEBUG: "false"
```

**Formato array:**

```yaml
environment:
  - SPRING_PROFILES_ACTIVE=production
  - DB_HOST=db
```

**Desde archivo `.env`:**

```yaml
services:
  app:
    env_file:
      - .env
      - .env.local
```

---

## 5.3. Interpolación de variables de entorno

**.env:**

```env
DB_PASSWORD=secreto123
DB_USER=appuser
APP_PORT=8080
JAVA_OPTS=-Xmx512m -Xms256m
```

**docker-compose.yml:**

```yaml
services:
  app:
    environment:
      DB_HOST: ${DB_HOST:-db}            # 'db' si no está definida
      DB_PASSWORD: ${DB_PASSWORD}        # Obligatoria (error si falta)
      JAVA_TOOL_OPTIONS: ${JAVA_OPTS}
    ports:
      - "${APP_PORT}:8080"

  db:
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_USER: ${DB_USER}
```

---

## 5.4. `volumes` (persistencia)

**Volumen nombrado** (Docker lo gestiona):

```yaml
services:
  db:
    volumes:
      - pg_data:/var/lib/postgresql/data

volumes:
  pg_data:
```

**Bind mount** (carpeta del host):

```yaml
services:
  app:
    volumes:
      - ./target:/app/target            # Relativo
      - /var/log/myapp:/var/log/myapp   # Absoluto
      - ./config:/app/config:ro         # Solo lectura
```

**Sintaxis larga (más control):**

```yaml
volumes:
  - type: bind
    source: ./config
    target: /app/config
    read_only: true
```

---

## 5.5. `networks` y segmentación

```yaml
services:
  nginx:
    networks:
      - frontend_net

  app:               # Spring Boot entre ambas redes
    networks:
      - frontend_net
      - backend_net

  db:
    networks:
      - backend_net  # Solo accesible desde app

networks:
  frontend_net:
  backend_net:
```

**Resultado:** `nginx` no puede hablar directamente con `db`. Seguridad por diseño.

```
nginx ←→ app ←→ db
(frontend)  (ambas)  (backend)
```

---

## 5.6. `deploy` — Límites de recursos

```yaml
services:
  app:
    deploy:
      resources:
        limits:
          cpus: '1.0'     # Máximo 1 CPU
          memory: 1G      # Máximo 1 GB RAM
        reservations:
          cpus: '0.5'     # Mínimo garantizado
          memory: 512M
```

> **Nota:** La sección `deploy` está orientada a Docker Swarm. Para limitar recursos en Compose local, también puedes usar `--cpus` y `--memory` en `docker run`.

# 6. Almacenamiento con Docker Compose

---

## 6. Tipos de almacenamiento

| Tipo | Uso | Persistencia |
|---|---|---|
| **Volumen nombrado** | Bases de datos, datos críticos | Sí (gestionado por Docker) |
| **Bind mount** | Código fuente en desarrollo | Sí (carpeta del host) |
| **tmpfs** | Datos temporales en memoria | No (se pierde al parar) |

---

## 6.1. Volúmenes nombrados

```yaml
services:
  db:
    image: postgres:15
    volumes:
      - pg_data:/var/lib/postgresql/data

  app:
    build: ./app
    volumes:
      - app_logs:/var/log/myapp

volumes:
  pg_data:    # Docker crea y gestiona este volumen
  app_logs:
```

```bash
# Listar volúmenes
$ docker volume ls

# Inspeccionar un volumen
$ docker volume inspect mi_proyecto_pg_data
```

---

## 6.2. Bind mounts para desarrollo Java

En desarrollo, puedes montar el `.jar` compilado directamente:

```yaml
services:
  app:
    image: eclipse-temurin:21-jre-alpine
    volumes:
      - ./target/myapp.jar:/app/app.jar:ro
    command: ["java", "-Dspring.profiles.active=dev", "-jar", "/app/app.jar"]
    ports:
      - "8080:8080"
```

O para desarrollo con live-reload usando `spring-boot-devtools`:

```yaml
services:
  app:
    build: ./app
    volumes:
      - ./src:/workspace/src             # Código fuente
      - ./target:/workspace/target       # Clases compiladas
    environment:
      SPRING_DEVTOOLS_RESTART_ENABLED: "true"
```

# 7. Redes con Docker Compose

---

## 7.1. Red predeterminada

Sin definir redes, Compose crea automáticamente una red `default`:

```yaml
services:
  web:
    image: nginx
  app:
    build: ./app
  db:
    image: postgres
# Los tres servicios pueden comunicarse entre sí por nombre
```

```bash
# La red se llama: <nombre_proyecto>_default
$ docker network ls
NETWORK ID     NAME                 DRIVER
abc123def456   mi_proyecto_default  bridge
```

---

## 7.2. Redes personalizadas (arquitectura en capas)

```yaml
services:
  nginx:
    image: nginx:alpine
    networks:
      - frontend

  app:                  # Spring Boot
    build: ./app
    networks:
      - frontend
      - backend

  db:
    image: postgres:15
    networks:
      - backend

  cache:
    image: redis:7
    networks:
      - backend

networks:
  frontend:
  backend:
```

# 8. Ejemplos prácticos con Java / Spring Boot

---

## 8.1. Spring Boot + PostgreSQL + Redis

```yaml
version: '3.8'

services:
  app:
    build: ./app
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/miapp
      SPRING_DATASOURCE_USERNAME: appuser
      SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD}
      SPRING_REDIS_HOST: cache
      SPRING_REDIS_PORT: 6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    networks:
      - backend

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: miapp
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pg_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d miapp"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - backend
```

---

## 8.1. Spring Boot + PostgreSQL + Redis (cont.)

```yaml
  cache:
    image: redis:7-alpine
    volumes:
      - redis_data:/data
    networks:
      - backend

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - app
    networks:
      - frontend
      - backend

volumes:
  pg_data:
  redis_data:

networks:
  frontend:
  backend:
```

---

## 8.1. Configuración Spring Boot — application.properties

```java
// Spring Boot usa automáticamente las variables de entorno
// application.properties
spring.datasource.url=${SPRING_DATASOURCE_URL}
spring.datasource.username=${SPRING_DATASOURCE_USERNAME}
spring.datasource.password=${SPRING_DATASOURCE_PASSWORD}
spring.redis.host=${SPRING_REDIS_HOST}
spring.redis.port=${SPRING_REDIS_PORT}
```

O con `application.yml`:

```yaml
spring:
  datasource:
    url: ${SPRING_DATASOURCE_URL:jdbc:postgresql://localhost:5432/miapp}
    username: ${SPRING_DATASOURCE_USERNAME:appuser}
    password: ${SPRING_DATASOURCE_PASSWORD}
  redis:
    host: ${SPRING_REDIS_HOST:localhost}
    port: ${SPRING_REDIS_PORT:6379}
```

> Spring Boot resuelve automáticamente las variables de entorno al arrancar.

---

## 8.2. Reintentar conexión en Java

`depends_on` solo garantiza que el contenedor de `db` **ha arrancado**, no que PostgreSQL esté aceptando conexiones. Spring Boot tiene reintentos configurables:

```java
// pom.xml — añade Spring Retry
// <dependency>
//   <groupId>org.springframework.retry</groupId>
//   <artifactId>spring-retry</artifactId>
// </dependency>

@SpringBootApplication
@EnableRetry
public class MiAplicacion {
    public static void main(String[] args) {
        SpringApplication.run(MiAplicacion.class, args);
    }
}
```

```yaml
# application.yml
spring:
  datasource:
    hikari:
      connection-timeout: 30000      # 30 seg esperando conexión
      initialization-fail-timeout: 60000  # 60 seg en startup
```

---

## 8.2. Migración de base de datos con Flyway

Ejecutar migraciones SQL como tarea puntual antes de arrancar la app:

```yaml
services:
  migrate:
    build: ./app
    command: ["java", "-jar", "app.jar", "--spring.flyway.enabled=true",
              "--spring.main.web-application-type=none"]
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/miapp
      SPRING_DATASOURCE_USERNAME: appuser
      SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD}
    depends_on:
      db:
        condition: service_healthy

  app:
    build: ./app
    depends_on:
      migrate:
        condition: service_completed_successfully
```

# 9. Buenas prácticas

---

## 9. Checklist de buenas prácticas

- ✅ Variables sensibles en `.env` (nunca en el YAML)
- ✅ `.env` en `.gitignore`
- ✅ Versiones específicas de imágenes (no `latest`)
- ✅ Nombres de servicios descriptivos
- ✅ Healthchecks en servicios críticos
- ✅ Límites de recursos definidos
- ✅ Configuración de logging
- ✅ Múltiples archivos para entornos
- ✅ Comentarios explicativos en el YAML
- ✅ README con instrucciones de uso

## 9.1. Gestión de secretos

**❌ MAL — nunca pongas contraseñas en el YAML:**

```yaml
services:
  db:
    environment:
      POSTGRES_PASSWORD: mi_password_secreto   # ¡Nunca hagas esto!
```

**✅ BIEN — usa archivo `.env`:**

```yaml
services:
  db:
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}
```

**.env (no versionado):**

```env
DB_PASSWORD=mi_password_secreto
```

**.gitignore:**

```
.env
```

---

## 9.1. Gestión de secretos — Docker Secrets (producción)

```yaml
services:
  db:
    image: postgres:15
    secrets:
      - db_password
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password

secrets:
  db_password:
    file: ./secrets/db_password.txt
```

```java
// En Java, leer el secreto desde el archivo montado:
public String readSecret(String secretName) throws IOException {
    Path secretPath = Path.of("/run/secrets/" + secretName);
    return Files.readString(secretPath).trim();
}
```

---

## 9.2. Versiones específicas de imágenes

**❌ MAL:**

```yaml
services:
  app:
    image: eclipse-temurin:latest    # ¿Qué versión exactamente?
  db:
    image: postgres:latest           # Puede romper la app sin avisar
```

**✅ BIEN:**

```yaml
services:
  app:
    build: ./app                     # Tu Dockerfile controla la versión
  db:
    image: postgres:15.4-alpine      # Versión exacta y reproducible
  cache:
    image: redis:7.2.3-alpine
```

> `latest` puede cambiar en cualquier momento, causar bugs difíciles de reproducir y entornos diferentes entre compañeros.

## 9.3. Nombres descriptivos y orden

**❌ MAL:**

```yaml
services:
  s1:
    image: nginx
  s2:
    image: postgres
  s3:
    build: ./app
```

**✅ BIEN — orden lógico de dependencias:**

```yaml
services:
  # 1. Frontend (expuesto al exterior)
  nginx:
    image: nginx:1.25.3
    depends_on: [api]

  # 2. Capa de aplicación
  api:
    build: ./app
    depends_on: [db, cache]

  # 3. Capa de datos (más internos)
  db:
    image: postgres:15.4-alpine
  cache:
    image: redis:7.2.3-alpine
```

## 9.4. Healthchecks

```yaml
services:
  db:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d miapp"]
      interval: 10s      # Cada cuánto verificar
      timeout: 5s        # Tiempo máximo de respuesta
      retries: 5         # Intentos antes de marcar unhealthy
      start_period: 30s  # Tiempo de gracia al inicio

  cache:
    image: redis:7
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3

  app:
    build: ./app
    depends_on:
      db:
        condition: service_healthy   # Espera a que db esté lista
      cache:
        condition: service_healthy
```

## 9.5. Límites de recursos y logging

**Límites de recursos:**

```yaml
services:

        limits:
          cpus: '1.0'
          memory: 1G
        reservations:
          cpus: '0.5'
          memory: 512M
```

**Logging para no llenar el disco:**

```yaml
services:
  app:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"   # Máximo 10 MB por archivo
        max-file: "3"     # Máximo 3 archivos rotados (30 MB total)
```

## 9.6. Múltiples entornos

**Estructura recomendada:**

```
proyecto/
├── docker-compose.yml          # Base común
├── docker-compose.dev.yml      # Desarrollo
├── docker-compose.prod.yml     # Producción
├── .env.example                # Plantilla de variables
└── .gitignore                  # Ignora .env
```

**docker-compose.dev.yml:**

```yaml
services:
  app:
    volumes:
      - ./src:/workspace/src   # Live reload
    environment:
      SPRING_PROFILES_ACTIVE: dev
      JAVA_TOOL_OPTIONS: -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005
    ports:
      - "8080:8080"
      - "5005:5005"            # Puerto de debugging remoto
```

---

## 9.6. Múltiples entornos (cont.)

**docker-compose.prod.yml:**

```yaml
services:
  app:
    restart: always
    environment:
      SPRING_PROFILES_ACTIVE: production
      JAVA_TOOL_OPTIONS: -Xmx1g -Xms512m
    ports:
      - "80:8080"
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "5"
```

**Uso:**

```bash
# Desarrollo
$ docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d

# Producción
$ docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

# Resumen

---

## Resumen

| Concepto | Clave |
|---|---|
| **Docker Compose** | Herramienta para apps multi-contenedor |
| **docker-compose.yml** | Define servicios, redes y volúmenes |
| **`docker compose up -d`** | Inicia toda la aplicación |
| **`docker compose down`** | Para y elimina contenedores y redes |
| **`depends_on`** | Orden de inicio (no espera readiness) |
| **Healthcheck** | Verifica que el servicio está listo |
| **Volumen nombrado** | Persistencia de datos gestionada por Docker |
| **Bind mount** | Mapeo de carpeta del host al contenedor |
| **`.env`** | Variables de entorno fuera del YAML |
| **Multi-archivo** | `docker-compose.dev.yml`, `prod.yml`... |
