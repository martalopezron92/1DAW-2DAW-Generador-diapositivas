## 1. El Problema del Código Espagueti

¿Qué pasa cuando el diseño es malo?

- Cambiar el formato de una fecha en la BD **requiere modificar 47 clases**
- Una clase `Usuario` también **envía emails, valida contraseñas, genera PDFs y se conecta a Twitter**
- No puedes probar un módulo sin **arrancar toda la aplicación**, incluyendo la BD y servicios externos

> Estos son síntomas claros de **alto acoplamiento** y **baja cohesión**.

---

## 1.1. El Equilibrio Ideal

**Objetivo del buen diseño de software:**

| Principio | Objetivo |
|---|---|
| 🔗 Bajo acoplamiento | Módulos independientes, evolución sin cascada |
| 🧲 Alta cohesión | Cada módulo tiene un propósito claro y único |
| 🛠 Mantenibilidad | Código fácil de entender y modificar |
| 🔄 Flexibilidad | Adaptarse sin refactorización masiva |
| 🧪 Testeabilidad | Cada módulo se prueba de forma aislada |
| ♻️ Reusabilidad | Módulos reutilizables en otros contextos |

<div class="tip">

**Principio clave**: Un buen diseño de software debe buscar **bajo acoplamiento** y **alta cohesión**.

</div>

---

## 1.2. Beneficios Tangibles

Cuando aplicas correctamente estos principios:

- 🐛 **Menos bugs**: cambios localizados → menos efectos colaterales
- ⚡ **Desarrollo más rápido**: código más fácil de entender y modificar
- 🧪 **Testing más simple**: módulos probables de forma independiente
- 👥 **Trabajo en equipo eficiente**: distintos desarrolladores en distintos módulos
- 📉 **Menor deuda técnica**: el código no se degrada con el tiempo
- 💰 **Menores costes de mantenimiento**: menos tiempo depurando, más tiempo aportando valor

---

## 2. Cohesión: Concepto Fundamental

**Cohesión** = Grado en que las responsabilidades de un módulo están relacionadas entre sí.

**Alta cohesión** significa que:
- Los elementos trabajan **hacia un objetivo común**
- La clase tiene una **responsabilidad clara y única**
- El nombre del módulo **describe perfectamente** todo lo que hace

**Baja cohesión** indica que:
- El módulo hace **múltiples cosas no relacionadas**
- Es difícil darle un nombre sin usar **"Y"** o **"Manager"**

<div class="tip">

**Regla práctica**: Si no puedes describir qué hace tu clase en una frase simple sin usar "Y", probablemente tiene baja cohesión.

</div>

---

## 2.1. Señales de Baja Cohesión

Identifica problemas cuando:

- ❌ La clase tiene más de 10-15 métodos públicos
- ❌ El nombre incluye `Manager`, `Handler`, `Util`, `Helper`
- ❌ La clase cambia frecuentemente por razones no relacionadas
- ❌ Los métodos usan diferentes subconjuntos de atributos
- ❌ Hay métodos estáticos que no usan ningún atributo de instancia
- ❌ Cuesta decidir dónde agregar nueva funcionalidad

---

## 2.2. Ejemplo Introductorio – Baja Cohesión ❌

```java
// ❌ BAJA COHESIÓN - Clase que hace de todo
public class Usuario {
    private String nombre;
    private String email;
    private String password;

    // Responsabilidad 1: Datos del usuario
    public void actualizarEmail(String nuevoEmail) { this.email = nuevoEmail; }

    // Responsabilidad 2: Autenticación
    public boolean validarPassword(String pwd) { return password.equals(pwd); }

    // Responsabilidad 3: Envío de emails
    public void enviarEmailBienvenida() {
        System.out.println("Enviando email a " + email);
    }

    // Responsabilidad 4: Persistencia
    public void guardarEnBaseDatos() {
        System.out.println("INSERT INTO usuarios...");
    }

    // Responsabilidad 5: Generación de reportes
    public byte[] generarReportePDF() { return new byte[0]; }

    // Responsabilidad 6: Logging
    public void registrarAccion(String accion) {
        System.out.println("[LOG] Usuario " + nombre + ": " + accion);
    }
}
```

**Problemas**: 6 responsabilidades · difícil testear · cambios en cascada

---

## 2.3. Ejemplo Introductorio – Alta Cohesión ✅

```java
// ✅ ALTA COHESIÓN - Responsabilidades separadas

// Solo datos y validación básica del dominio
public class Usuario {
    private int id;
    private String nombre;
    private String email;

    public void actualizarEmail(String nuevoEmail) {
        if (!nuevoEmail.contains("@")) throw new IllegalArgumentException("Email inválido");
        this.email = nuevoEmail;
    }
    // getters...
}

// Solo autenticación
public class ServicioAutenticacion {
    public boolean validarCredenciales(String email, String password) { return true; }
}

// Solo emails
public class ServicioEmail {
    public void enviarBienvenida(Usuario usuario) {
        System.out.println("Enviando bienvenida a " + usuario.getEmail());
    }
}

// Solo persistencia
public class RepositorioUsuarios {
    public void guardar(Usuario usuario) { System.out.println("INSERT INTO usuarios..."); }
    public Usuario buscarPorEmail(String email) { return null; }
}
```

## 3. Tipos de Cohesión

| Nivel | Tipo | Calidad | Esfuerzo de mantenimiento |
|---|---|---|---|
| 1 | Coincidental | ❌ La peor | Muy alto |
| 2 | Lógica | 🔴 Muy baja | Alto |
| 3 | Temporal | 🟠 Baja | Alto |
| 4 | Procedimental | 🟡 Media-baja | Medio-alto |
| 5 | Comunicacional | 🟢 Media | Medio |
| 6 | Secuencial | 🔵 Alta | Bajo |
| 7 | **Funcional** | ⭐ **La mejor** | Muy bajo |

<div class="warn">

**Objetivo**: Aspira siempre a **cohesión funcional** (nivel 7) o como mínimo a **cohesión secuencial** (nivel 6). Los niveles 1-4 indican serios problemas de diseño.

</div>

---

## 3.1. Cohesión Coincidental (Nivel 1 ❌)

Elementos agrupados **sin ninguna relación aparente**. La temida clase `Utilidades`.

```java
// ❌ COHESIÓN COINCIDENTAL
public class Utilidades {
    public double calcularIva(double precio) { return precio * 0.21; }

    public void enviarEmail(String destinatario, String mensaje) {
        System.out.println("Enviando email a " + destinatario + ": " + mensaje);
    }

    public List<Integer> ordenarArray(List<Integer> lista) {
        return lista.stream().sorted().collect(Collectors.toList());
    }

    public String obtenerFechaActual() {
        return LocalDate.now().toString();
    }

    public boolean esEmailValido(String email) { return email.contains("@"); }

    public String generarId() { return UUID.randomUUID().toString(); }
}
```

**Problemas**: No puedes describir la clase · agregar funcionalidad es arbitrario · testing difícil

---

## 3.1. Solución: Separar por Dominio ✅

```java
// ✅ Alta cohesión - cada clase con propósito claro

public class CalculadoraPrecios {
    private static final double TASA_IVA = 0.21;
    public double calcularIva(double precio) { return precio * TASA_IVA; }
    public double calcularPrecioFinal(double precioBase) {
        return precioBase + calcularIva(precioBase);
    }
}

public class ServicioEmail {
    public void enviar(String destinatario, String mensaje) {
        System.out.println("Enviando email a " + destinatario + ": " + mensaje);
    }
}

public class ValidadorEmail {
    public boolean esValido(String email) {
        return email.matches("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$");
    }
}

public class GeneradorIdentificadores {
    public String generarUUID() { return UUID.randomUUID().toString(); }
    public long generarIdNumerico() { return System.currentTimeMillis(); }
}
```

## 3.2. Cohesión Lógica (Nivel 2 🔴)

Elementos relacionados por **tipo**, no por función. Controlados por un parámetro que selecciona la operación.

```java
// ❌ COHESIÓN LÓGICA - Operaciones del mismo tipo pero no relacionadas
public class Operaciones {
    public int ejecutar(String tipo, List<Integer> datos) {
        switch (tipo) {
            case "SUMA":    return datos.stream().mapToInt(Integer::intValue).sum();
            case "PROMEDIO":
                return datos.isEmpty() ? 0 :
                       datos.stream().mapToInt(Integer::intValue).sum() / datos.size();
            case "MAXIMO":
                return datos.stream().mapToInt(Integer::intValue).max().orElse(0);
            default: return 0;
        }
    }
}

// Uso confuso - ¿qué hace "ejecutar"?
Operaciones ops = new Operaciones();
int suma     = ops.ejecutar("SUMA", Arrays.asList(1, 2, 3));
int maximo   = ops.ejecutar("MAXIMO", Arrays.asList(1, 2, 3));
```

**Viola el Principio Abierto/Cerrado** · difícil extender · testing complejo

---

## 3.2. Solución: Interfaz + Clases Especializadas ✅

```java
// ✅ Cohesión funcional - Operaciones separadas con interfaz común
public interface OperacionNumerica {
    int ejecutar(List<Integer> datos);
}

public class OperacionSuma implements OperacionNumerica {
    @Override
    public int ejecutar(List<Integer> datos) {
        return datos.stream().mapToInt(Integer::intValue).sum();
    }
}

public class OperacionPromedio implements OperacionNumerica {
    @Override
    public int ejecutar(List<Integer> datos) {
        return datos.isEmpty() ? 0 :
               datos.stream().mapToInt(Integer::intValue).sum() / datos.size();
    }
}

public class OperacionMaximo implements OperacionNumerica {
    @Override
    public int ejecutar(List<Integer> datos) {
        return datos.stream().mapToInt(Integer::intValue).max().orElse(0);
    }
}

// Uso claro y extensible (nuevo operación = nueva clase)
int suma     = new OperacionSuma().ejecutar(Arrays.asList(1, 2, 3));
int maximo   = new OperacionMaximo().ejecutar(Arrays.asList(1, 2, 3));
```

## 3.3. Cohesión Temporal (Nivel 3 🟠)

Elementos agrupados porque se ejecutan en el **mismo momento**, pero sin relación funcional.

```java
// ❌ COHESIÓN TEMPORAL - Todo se ejecuta al inicio, pero no está relacionado
public class Aplicacion {
    public void inicializar() {
        // 1. Configurar logging
        System.setProperty("log.level", "INFO");
        System.out.println("Logger inicializado");

        // 2. Conectar base de datos
        conectarBaseDatos();

        // 3. Cargar configuración
        Map<String, String> config = cargarConfiguracion();

        // 4. Inicializar caché
        inicializarCache();

        // 5. Registrar servicios
        registrarServicios();

        // 6. Verificar licencia
        verificarLicencia();
    }
    // ...métodos privados sin relación entre sí
}
```

**Problemas**: Si falla uno fallan todos · no puedes inicializar solo una parte · difícil testear

---

## 3.3. Solución: Inicializadores Separados ✅

```java
// ✅ Alta cohesión - Cada inicializador tiene su responsabilidad
public interface Inicializador {
    void inicializar();
    String nombre();
}

public class InicializadorLogger implements Inicializador {
    @Override public void inicializar() { System.setProperty("log.level", "INFO"); }
    @Override public String nombre() { return "Logger"; }
}

public class InicializadorBaseDatos implements Inicializador {
    @Override public void inicializar() { System.out.println("Base de datos conectada"); }
    @Override public String nombre() { return "Base de Datos"; }
}

// Coordinador que maneja el orden y los errores
public class GestorInicializacion {
    private final List<Inicializador> inicializadores = new ArrayList<>();

    public void agregar(Inicializador ini) { inicializadores.add(ini); }

    public void inicializarTodo() {
        for (Inicializador ini : inicializadores) {
            try {
                System.out.println("Iniciando " + ini.nombre() + "...");
                ini.inicializar();
                System.out.println("✓ " + ini.nombre() + " listo");
            } catch (Exception e) {
                System.out.println("✗ Error en " + ini.nombre() + ": " + e.getMessage());
            }
        }
    }
}
```

## 3.4. Cohesión Procedimental (Nivel 4 🟡)

Elementos relacionados por **secuencia de pasos**, pero cada paso trabaja con datos diferentes.

```java
// ❌ COHESIÓN PROCEDIMENTAL - Procesamiento de pedido en un solo método
public class ProcesadorPedido {
    public boolean procesarPedido(Pedido pedido) {
        // Paso 1: Validar datos del cliente
        if (!validarDatosCliente(pedido.getCliente())) {
            System.out.println("Cliente inválido"); return false;
        }
        // Paso 2: Verificar stock
        if (!verificarDisponibilidadStock(pedido.getItems())) {
            System.out.println("Stock insuficiente"); return false;
        }
        // Paso 3: Calcular precio total
        double total = calcularPrecioTotal(pedido.getItems());

        // Paso 4: Procesar pago
        if (!procesarPago(pedido.getCliente(), total)) {
            System.out.println("Pago rechazado"); return false;
        }
        // Paso 5: Actualizar inventario
        actualizarInventario(pedido.getItems());

        // Paso 6: Enviar confirmación
        enviarEmailConfirmacion(pedido.getCliente().getEmail(), pedido);
        return true;
    }
    // Múltiples métodos privados con responsabilidades muy distintas...
}
```

**Viola SRP** · mezcla validación, cálculo, comunicación y persistencia

---

## 3.4. Solución: Separar por Dominio y Componer ✅

```java
// ✅ Alta cohesión - cada componente con una responsabilidad clara

public class ValidadorPedido {
    public ResultadoValidacion validarCliente(Cliente cliente) {
        List<String> errores = new ArrayList<>();
        if (cliente.getNombre().isEmpty()) errores.add("Nombre requerido");
        if (!cliente.getEmail().matches("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$"))
            errores.add("Email inválido");
        return errores.isEmpty() ? ResultadoValidacion.exitoso()
                                 : ResultadoValidacion.fallido(errores);
    }
}

public class CalculadoraPrecios {
    public double calcularTotalPedido(List<ItemPedido> items) {
        return items.stream().mapToDouble(i -> i.getPrecio() * i.getCantidad()).sum();
    }
    public double aplicarDescuento(double total, double porcentaje) {
        return total * (1 - porcentaje / 100);
    }
    public double calcularImpuestos(double total) { return total * 0.21; }
}

public class ServicioPagos {
    public ResultadoPago procesar(Cliente cliente, double monto) {
        System.out.println("Procesando pago de " + monto + " para " + cliente.getNombre());
        return monto > 0 ? ResultadoPago.aprobado("TX-" + System.currentTimeMillis())
                         : ResultadoPago.rechazado("Monto inválido");
    }
}
```

## 3.5. Cohesión Comunicacional (Nivel 5 🟢)

Elementos relacionados porque **operan sobre los mismos datos de entrada/salida**, aunque hacen cosas distintas con ellos.

```java
// ⚠️ COHESIÓN COMUNICACIONAL - Operaciones diferentes sobre los mismos datos
public class ProcesadorArchivo {
    public void procesarArchivo(String nombreArchivo) {
        String contenido = leerArchivo(nombreArchivo);

        int numLineas  = contarLineas(contenido);
        int numPalabras = contarPalabras(contenido);
        int ocurrencias = buscarPatron(contenido, "error");
        String stats   = calcularEstadisticas(contenido);

        System.out.println("Líneas: " + numLineas);
        System.out.println("Palabras: " + numPalabras);
        System.out.println("Errores: " + ocurrencias);
        System.out.println(stats);
    }
    // Todos los métodos reciben 'contenido' pero hacen cosas distintas
}
```

**Problema**: No puedes usar una operación sin las otras · difícil extender

---

## 3.5. Solución: Clases Analizadoras Independientes ✅

```java
// ✅ Alta cohesión - cada analizador con su propósito específico

public class ContadorLineas {
    public int contar(String texto) { return texto.split("\n", -1).length; }
    public int contarNoVacias(String texto) {
        return (int) Arrays.stream(texto.split("\n")).filter(l -> !l.isBlank()).count();
    }
}

public class BuscadorPatrones {
    public int buscar(String texto, String patron) {
        return (int) Pattern.compile(patron, Pattern.CASE_INSENSITIVE)
                            .matcher(texto).results().count();
    }
    public Map<String, Integer> buscarMultiples(String texto, List<String> patrones) {
        return patrones.stream().collect(Collectors.toMap(p -> p, p -> buscar(texto, p)));
    }
}

public class CalculadorEstadisticas {
    public Estadisticas calcular(String texto) {
        ContadorLineas cl = new ContadorLineas();
        ContadorPalabras cp = new ContadorPalabras();
        return new Estadisticas(texto.length(), cl.contar(texto),
                                cl.contarNoVacias(texto), cp.contar(texto));
    }
}
// Cada herramienta es independiente y reutilizable
```

## 3.6. Cohesión Secuencial (Nivel 6 🔵)

La **salida de un elemento es la entrada del siguiente**, formando una cadena de procesamiento.

```java
// ✅ COHESIÓN SECUENCIAL - Pipeline con API fluente (Builder pattern)
public class PipelineImagen {
    private final String rutaOrigen;
    private byte[] datos = new byte[0];

    public PipelineImagen(String rutaOrigen) { this.rutaOrigen = rutaOrigen; }

    public PipelineImagen cargar() {
        System.out.println("Cargando " + rutaOrigen);
        this.datos = new byte[]{1, 2, 3}; // simulado
        return this;
    }

    public PipelineImagen redimensionar(int ancho, int alto) {
        System.out.println("Redimensionando a " + ancho + "x" + alto);
        return this;
    }

    public PipelineImagen aplicarFiltro(String filtro) {
        System.out.println("Aplicando filtro " + filtro);
        return this;
    }

    public PipelineImagen comprimir(int calidad) {
        System.out.println("Comprimiendo (calidad: " + calidad + ")");
        return this;
    }

    public boolean guardar(String rutaDestino) {
        System.out.println("Guardando en " + rutaDestino);
        return true;
    }
}

// Uso fluente encadenado
new PipelineImagen("foto.jpg")
    .cargar().redimensionar(800, 600).aplicarFiltro("SEPIA").comprimir(85)
    .guardar("foto_procesada.jpg");
```

## 3.7. Cohesión Funcional (Nivel 7 ⭐)

**Todos los elementos contribuyen a una única tarea bien definida.** Es el nivel que debemos buscar.

```java
// ✅ COHESIÓN FUNCIONAL - Validador de email con responsabilidad única
public class ValidadorEmail {
    private static final Pattern PATRON_EMAIL =
        Pattern.compile("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$");

    public boolean esValido(String email) {
        return PATRON_EMAIL.matcher(email).matches();
    }

    public ResultadoValidacion validar(String email) {
        if (email == null || email.isBlank())
            return ResultadoValidacion.invalido("Email no puede estar vacío");
        if (!email.contains("@"))
            return ResultadoValidacion.invalido("Email debe contener @");
        if (!esValido(email))
            return ResultadoValidacion.invalido("Formato de email inválido");
        return ResultadoValidacion.valido();
    }

    public Optional<String> extraerDominio(String email) {
        return esValido(email) ? Optional.of(email.substring(email.indexOf('@') + 1))
                               : Optional.empty();
    }
}
```

**Una responsabilidad · todos los métodos relacionados con emails · nombre descriptivo · fácil testear**

## 4. Acoplamiento: Concepto Fundamental

**Acoplamiento** = Grado de interdependencia entre módulos.

**Bajo acoplamiento** implica:
- Cambios en un módulo **no requieren cambios en otros**
- Módulos pueden **evolucionar de forma independiente**
- Cada módulo puede **testearse de forma aislada**

**Alto acoplamiento** implica:
- Cambios en cascada — modificar A rompe B, C, D...
- Testing costoso — necesitas instanciar medio proyecto
- Código frágil y difícil de mantener

<div class="warn">

**Nota**: Es imposible tener acoplamiento cero (los módulos deben comunicarse). El objetivo es **minimizarlo y hacerlo explícito**.

</div>

---

## 4.1. Señales de Alto Acoplamiento

- ❌ Cambiar una clase requiere cambiar 5-10 clases más
- ❌ No puedes testear una clase sin instanciar otras 10
- ❌ Una clase accede directamente a atributos de otra
- ❌ Muchas clases importan la misma clase concreta
- ❌ Cambios en la BD rompen la UI
- ❌ No puedes reutilizar un módulo sin arrastrar medio proyecto

---

## 4.2. Ejemplo Introductorio – Alto Acoplamiento ❌

```java
// ❌ ALTO ACOPLAMIENTO - Dependencias directas y rígidas
public class ProcesadorPedido {
    private BaseDatosMySQL baseDatos = new BaseDatosMySQL();      // Concreta
    private ServidorEmailSMTP emailSender = new ServidorEmailSMTP(); // Concreta
    private LoggerArchivo logger = new LoggerArchivo();             // Concreta

    public void procesar(Pedido pedido) {
        baseDatos.conectar("localhost", "user", "password");
        baseDatos.guardar("INSERT INTO pedidos...");

        emailSender.configurar("smtp.gmail.com", 587);
        emailSender.enviar("cliente@example.com", "Pedido confirmado");

        logger.escribir("/var/log/app.log", "Pedido: " + pedido.getId());
    }
}
```

**Problemas**:
- Cambiar de MySQL a PostgreSQL → modificar `ProcesadorPedido`
- No puedes testear sin una BD real, SMTP real, etc.
- Conoce detalles de implementación (hosts, puertos, rutas)

---

## 4.2. Ejemplo Introductorio – Bajo Acoplamiento ✅

```java
// ✅ BAJO ACOPLAMIENTO - Dependencias mediante interfaces
public interface RepositorioPedidos { void guardar(Pedido pedido); }
public interface ServicioEmail { void enviar(String dest, String msg); }
public interface Logger { void log(String mensaje); }

public class ProcesadorPedido {
    private final RepositorioPedidos repositorio;
    private final ServicioEmail emailService;
    private final Logger logger;

    public ProcesadorPedido(RepositorioPedidos repo, ServicioEmail email, Logger logger) {
        this.repositorio = repo;
        this.emailService = email;
        this.logger = logger;
    }

    public void procesar(Pedido pedido) {
        repositorio.guardar(pedido);
        emailService.enviar(pedido.getCliente().getEmail(), "Pedido confirmado");
        logger.log("Pedido procesado: " + pedido.getId());
    }
}

// Uso con inyección de dependencias
ProcesadorPedido procesador = new ProcesadorPedido(
    new RepositorioPedidosMySQL(),
    new ServicioEmailSMTP(),
    new LoggerConsola()
);
```

## 5. Tipos de Acoplamiento

| Nivel | Tipo | Calidad |
|---|---|---|
| 1 | Contenido | ❌ La peor |
| 2 | Común | 🔴 Muy malo |
| 3 | Externo | 🟠 Malo |
| 4 | Control | 🟡 Regular |
| 5 | Estampado (Stamp) | 🟢 Aceptable |
| 6 | Datos | 🔵 Bueno |
| 7 | **Mensaje** | ⭐ **La mejor** |

---

## 5.1. Acoplamiento de Contenido (Nivel 1 ❌)

Un módulo **modifica o accede directamente al interior de otro** (variables privadas, flujo interno).

```java
// ❌ ACOPLAMIENTO DE CONTENIDO - Acceso a privados mediante reflexión
public class HackerCuenta {
    public void modificarSaldoDirectamente(Usuario usuario, double nuevoSaldo)
            throws Exception {
        Field campo = Usuario.class.getDeclaredField("saldo");
        campo.setAccessible(true);
        campo.setDouble(usuario, nuevoSaldo);
        System.out.println("Saldo modificado directamente: " + nuevoSaldo);
    }
}
```

**Solución**: Respetar encapsulación y usar solo la API pública:

```java
// ✅ BAJO ACOPLAMIENTO - Solo API pública
public class Usuario {
    private double saldo;

    public boolean depositar(double monto) {
        if (monto > 0) { saldo += monto; return true; }
        return false;
    }
    public boolean retirar(double monto) {
        if (monto > 0 && saldo >= monto) { saldo -= monto; return true; }
        return false;
    }
    public double obtenerSaldo() { return saldo; }
}
```

---

## 5.2. Acoplamiento Común (Nivel 2 🔴)

Módulos **comparten datos globales**. Cualquiera puede leer y modificar.

```java
// ❌ ACOPLAMIENTO COMÚN - Estado global compartido (Singleton mutable)
public class ConfiguracionGlobal {
    public static String nombreUsuario = "";
    public static String idioma = "es";
    public static double tasaImpuesto = 0.21;
}

public class ServicioFacturacion {
    public double generarFactura(double monto) {
        return monto + (monto * ConfiguracionGlobal.tasaImpuesto); // Dependencia global
    }
}
```

**Solución**: Inyectar la configuración como dependencia:

```java
// ✅ Configuración inyectada
public class Configuracion {
    private final String idioma;
    private final double tasaImpuesto;
    public Configuracion(String idioma, double tasaImpuesto) {
        this.idioma = idioma; this.tasaImpuesto = tasaImpuesto;
    }
    // getters...
}

public class ServicioFacturacion {
    private final Configuracion config;
    public ServicioFacturacion(Configuracion config) { this.config = config; }
    public double generarFactura(double monto) {
        return monto + (monto * config.getTasaImpuesto());
    }
}
```

## 5.3. Acoplamiento Externo (Nivel 3 🟠)

Módulos dependen de un **formato o protocolo externo compartido**.

```java
// ❌ Formato CSV hardcodeado en múltiples clases
public class LectorUsuarios {
    public List<Usuario> leer(String csv) {
        return Arrays.stream(csv.split("\n")).map(linea -> {
            String[] partes = linea.split(",");
            return new Usuario(partes[1], partes[2]); // Conoce el formato
        }).collect(Collectors.toList());
    }
}

public class ValidadorUsuarios {
    public boolean validar(String csv) {
        return Arrays.stream(csv.split("\n")).allMatch(linea -> {
            String[] partes = linea.split(",");
            return partes.length == 3 && partes[2].contains("@"); // También conoce el formato
        });
    }
}
```

**Solución**: Encapsular el formato en un parser centralizado:

```java
// ✅ Parser centralizado
public class ParserCSVUsuarios {
    public List<Usuario> parsear(String csv) { /* única implementación del formato */ }
    public String serializar(List<Usuario> usuarios) { /* única impl. de serialización */ }
}
// Lectores y validadores usan solo objetos Usuario, no el formato CSV
```

## 5.4. Acoplamiento de Control (Nivel 4 🟡)

Un módulo controla el **flujo interno de otro** mediante flags o parámetros de comportamiento.

```java
// ❌ ACOPLAMIENTO DE CONTROL - Flag booleano que controla comportamiento interno
public class ServicioEmail {
    public void enviar(String destinatario, String mensaje,
                       boolean urgente, boolean conCopia) {
        if (urgente) { /* configuración urgente */ }
        if (conCopia) { /* añadir copia a admin */ }
        System.out.println("Enviando a " + destinatario);
    }
}

// Llamadas confusas
servicio.enviar("user@ex.com", "Hola", false, false);
servicio.enviar("user@ex.com", "¡Urgente!", true, true);
```

**Solución**: Encapsular el comportamiento en objetos de configuración:

```java
// ✅ Comportamiento encapsulado en estrategias
public interface ConfiguracionEmail {
    void aplicar(EmailBuilder builder);
}

public class ConfiguracionEmailUrgente implements ConfiguracionEmail {
    @Override
    public void aplicar(EmailBuilder builder) {
        builder.setPrioridad(Prioridad.URGENTE);
        builder.setCopias(Arrays.asList("admin@empresa.com"));
    }
}
```

## 5.5. Acoplamiento de Estampado / Stamp (Nivel 5 🟢)

Se pasa un **objeto completo** cuando solo se necesitan algunos de sus campos.

```java
// ⚠️ ACOPLAMIENTO DE ESTAMPADO - Objeto completo innecesario
public class CalculadoraEnvio {
    // Solo necesita la dirección, pero recibe el pedido completo
    public double calcularCosto(Pedido pedido) {
        String ciudad = pedido.getDireccionEnvio().getCiudad();
        if (ciudad.equals("Madrid")) return 5.0;
        if (ciudad.equals("Barcelona")) return 7.0;
        return 10.0;
    }
}
```

**Solución**: Pasar solo lo necesario:

```java
// ✅ Pasar solo los datos necesarios
public class CalculadoraEnvio {
    public double calcularCosto(Direccion direccion) {
        if (direccion.getCiudad().equals("Madrid")) return 5.0;
        if (direccion.getCiudad().equals("Barcelona")) return 7.0;
        return 10.0;
    }
}

// Uso explícito y claro
CalculadoraEnvio calculadora = new CalculadoraEnvio();
double coste = calculadora.calcularCosto(pedido.getDireccionEnvio());
```

## 5.6. Acoplamiento de Datos (Nivel 6 🔵)

Módulos se comunican mediante **parámetros de datos simples** (primitivos, strings, listas).

```java
// ✅ ACOPLAMIENTO DE DATOS - Parámetros simples y claros
public class Calculadora {
    public int sumar(int a, int b) { return a + b; }
    public double calcularPorcentaje(double valor, double porcentaje) {
        return valor * (porcentaje / 100);
    }
}

public class ValidadorTexto {
    public boolean esEmailValido(String email) {
        return email.matches("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$");
    }
    public boolean longitudValida(String texto, int minimo, int maximo) {
        return texto.length() >= minimo && texto.length() <= maximo;
    }
}

public class FormateadorFecha {
    public String formatear(int anio, int mes, int dia) {
        return String.format("%04d-%02d-%02d", anio, mes, dia);
    }
}
```

**Ventajas**: Completamente desacoplado · fácil testear · reutilizable en cualquier contexto

## 5.7. Acoplamiento de Mensaje (Nivel 7 ⭐)

Comunicación únicamente mediante **interfaces**, sin conocer implementaciones concretas.

```java
// ✅ ACOPLAMIENTO DE MENSAJE - Solo interfaces
public interface RepositorioUsuarios {
    void guardar(Usuario usuario);
    Optional<Usuario> buscarPorEmail(String email);
    List<Usuario> listarTodos();
}

public interface Notificador {
    void notificar(String destinatario, String mensaje);
}

// Servicio que SOLO depende de interfaces
public class ServicioRegistro {
    private final RepositorioUsuarios repositorio;
    private final Notificador notificador;

    public ServicioRegistro(RepositorioUsuarios repo, Notificador notificador) {
        this.repositorio = repo;
        this.notificador = notificador;
    }

    public boolean registrar(Usuario usuario) {
        repositorio.guardar(usuario);
        notificador.notificar(usuario.getEmail(), "Bienvenido " + usuario.getNombre());
        return true;
    }
}

// Implementaciones intercambiables sin modificar ServicioRegistro
ServicioRegistro s1 = new ServicioRegistro(new RepositorioMemoria(), new NotificadorEmail());
ServicioRegistro s2 = new ServicioRegistro(new RepositorioMemoria(), new NotificadorSMS());
```

## 6. Estrategias para Reducir el Acoplamiento

### 6.1. Inyección de Dependencias (DI)

```java
// ❌ Sin DI - Alto acoplamiento
public class Servicio {
    private RepositorioMySQL repositorio = new RepositorioMySQL(); // Dependencia concreta

    public void hacer() { repositorio.guardar(); }
}

// ✅ Con DI - Bajo acoplamiento
public class Servicio {
    private final Repositorio repositorio; // Interfaz

    public Servicio(Repositorio repositorio) { this.repositorio = repositorio; }

    public void hacer() { repositorio.guardar(); }
}
```

## 6.2. Principio de Inversión de Dependencias (DIP)

```java
// ❌ Depende de concreción
public class Notificador {
    public void notificar(ServidorEmailSMTP email, String msg) {
        email.enviar(msg);
    }
}

// ✅ Depende de abstracción
public interface ServicioEmail {
    void enviar(String mensaje);
}

public class Notificador {
    private final ServicioEmail servicioEmail;

    public Notificador(ServicioEmail servicioEmail) {
        this.servicioEmail = servicioEmail;
    }

    public void notificar(String mensaje) {
        servicioEmail.enviar(mensaje);
    }
}
```

## 6.3. Ley de Demeter

No hables con extraños, solo con amigos directos.

```java
// ❌ Viola Ley de Demeter - Cadena de llamadas larga
public class GestorPago {
    public void procesarPago(Pedido pedido) {
        double monto = pedido.getCarrito().getItems().stream()
                             .mapToDouble(Item::getPrecio).sum(); // 3 niveles
        System.out.println("Procesando " + monto);
    }
}

// ✅ Respeta Ley de Demeter - Solo un nivel
public class Pedido {
    public double calcularTotal() {
        return carrito.calcularTotal(); // Pedido delega a Carrito
    }
}

public class GestorPago {
    public void procesarPago(Pedido pedido) {
        double monto = pedido.calcularTotal(); // Un único nivel
        System.out.println("Procesando " + monto);
    }
}
```

## 6.4. Patrón Facade

Simplifica interfaces complejas para reducir el acoplamiento del cliente.

```java
// Sistema con subsistemas complejos
public class SubsistemaA { public void operacion1() {} }
public class SubsistemaB { public void operacion2() {} }
public class SubsistemaC { public void operacion3() {} }

// Facade simplifica el acceso
public class FachadeSistema {
    private final SubsistemaA a = new SubsistemaA();
    private final SubsistemaB b = new SubsistemaB();
    private final SubsistemaC c = new SubsistemaC();

    public void operacionSimplificada() {
        a.operacion1();
        b.operacion2();
        c.operacion3();
    }
}

// El cliente solo conoce la fachada (bajo acoplamiento)
public class Cliente {
    private final FachadeSistema fachada;
    public Cliente(FachadeSistema fachada) { this.fachada = fachada; }
    public void hacer() { fachada.operacionSimplificada(); }
}
```

## 7. La Relación entre Cohesión y Acoplamiento

### Matriz de calidad del diseño

|  | Bajo Acoplamiento | Alto Acoplamiento |
|---|---|---|
| **Alta Cohesión** | ⭐ **EXCELENTE** — Módulos independientes con propósito claro. Fácil mantener, testear y reutilizar. | ⚠️ **BUENO** — Módulos cohesivos pero dependientes. Viable pero complejo. |
| **Baja Cohesión** | ⚠️ **REGULAR** — Módulos independientes pero confusos. Difícil entender. | ❌ **MALO** — Código espagueti. Confuso y dependiente. Imposible mantener. |

<div class="tip">

**Regla práctica**: Prioriza siempre la alta cohesión sobre el bajo acoplamiento. Es mejor tener módulos cohesivos ligeramente acoplados que módulos desacoplados pero confusos.

</div>

## 8. Principios SOLID Relacionados

### 8.1. SRP – Principio de Responsabilidad Única

Una clase debe tener una **sola razón para cambiar** → promueve directamente la **alta cohesión**.

```java
// ✅ SRP + Alta Cohesión
public class ValidadorEmail {
    public boolean esValido(String email) { return email.contains("@"); }
    public String extraerDominio(String email) {
        int idx = email.indexOf('@');
        return idx >= 0 ? email.substring(idx + 1) : "";
    }
}
```

### 8.2. DIP – Principio de Inversión de Dependencias

Depender de **abstracciones**, no de concreciones → reduce directamente el **acoplamiento**.

```java
// ✅ DIP + Bajo Acoplamiento
public interface Repositorio { void guardar(Object dato); }

public class Servicio {
    private final Repositorio repositorio; // Abstracción, no concreción
    public Servicio(Repositorio repositorio) { this.repositorio = repositorio; }
    public void procesar(Object dato) { repositorio.guardar(dato); }
}
```

---

## 8.3. ISP – Principio de Segregación de Interfaces

Los clientes no deben depender de interfaces que no usan → reduce el **acoplamiento innecesario**.

```java
// ❌ Interfaz "gorda" - acoplamiento innecesario
public interface Trabajador {
    void trabajar();
    void comer();    // Los robots no comen
    void dormir();   // Los robots no duermen
}

public class Robot implements Trabajador {
    public void trabajar() { /* OK */ }
    public void comer()   { /* ¡No aplica! ❌ */ }
    public void dormir()  { /* ¡No aplica! ❌ */ }
}

// ✅ Interfaces segregadas
public interface Trabajador { void trabajar(); }
public interface SerVivo { void comer(); void dormir(); }

public class Robot implements Trabajador {
    public void trabajar() { /* OK */ }
}

public class Humano implements Trabajador, SerVivo {
    public void trabajar() { /* OK */ }
    public void comer()    { /* OK */ }
    public void dormir()   { /* OK */ }
}
```

## 9. Métricas: LCOM (Falta de Cohesión de Métodos)

**LCOM** = pares de métodos que **NO** comparten atributos − pares que **SÍ** comparten atributos.

- **LCOM = 0**: Perfecta cohesión
- **LCOM bajo (< 50)**: Buena cohesión
- **LCOM alto (> 100)**: Baja cohesión → considera dividir la clase

```java
// ✅ LCOM bajo - Alta cohesión (todos los métodos usan los mismos atributos)
public class Rectangulo {
    private double ancho;
    private double alto;

    public double calcularArea()      { return ancho * alto; }
    public double calcularPerimetro() { return 2 * (ancho + alto); }
    public void escalar(double factor) { ancho *= factor; alto *= factor; }
}

// ❌ LCOM alto - Baja cohesión (cada método usa atributos distintos)
public class Utilidades {
    private int contador;
    private String email;

    public void incrementar()       { contador++; }           // solo 'contador'
    public boolean validarEmail()   { return email.contains("@"); } // solo 'email'
    // Ningún método usa ambos atributos → LCOM alto
}
```

## 9.1. Métricas de Acoplamiento

**Acoplamiento Aferente (Ca)**: nº de clases externas que **dependen de** esta clase.
- Ca alto → clase estable, muchas la usan → menos propensa a cambios.

**Acoplamiento Eferente (Ce)**: nº de clases externas de las que **depende** esta clase.
- Ce alto → clase dependiente, usa muchas otras.

**Inestabilidad (I)**:
```
I = Ce / (Ca + Ce)
```
- **I = 0** → clase máximamente estable
- **I = 1** → clase máximamente inestable

```
Ejemplo Clase A:  Ca=5, Ce=1  →  I = 1/(5+1) = 0.16  →  Muy estable ✅
Ejemplo Clase B:  Ca=1, Ce=10 →  I = 10/(1+10) = 0.91 →  Muy inestable ⚠️
```

**Herramientas**: SonarQube · IntelliJ IDEA · PMD · JDepend

```java
public class Producto { private int id; private String nombre; private double precio; /* ... */ }
```

## 10. Caso Práctico: Orquestador con Dependencias Inyectadas ✅

```java
// ✅ Orquestador: coordina pero no implementa detalles
public class ServicioPedidos {
    private final RepositorioClientes repoClientes;
    private final RepositorioProductos repoProductos;
    private final RepositorioPedidos repoPedidos;
    private final ValidadorPedido validador;
    private final CalculadoraPrecios calculadora;
    private final ServicioNotificaciones notificaciones;

    public ServicioPedidos(RepositorioClientes rc, RepositorioProductos rp,
                           RepositorioPedidos rpd, ValidadorPedido val,
                           CalculadoraPrecios calc, ServicioNotificaciones notif) {
        this.repoClientes = rc; this.repoProductos = rp; this.repoPedidos = rpd;
        this.validador = val; this.calculadora = calc; this.notificaciones = notif;
    }

    public Optional<Pedido> procesarPedido(int clienteId, List<Integer> productosIds) {
        Optional<Cliente> cliente = repoClientes.buscarPorId(clienteId);
        if (!validador.validarCliente(cliente.orElse(null))) return Optional.empty();

        List<ItemPedido> items = productosIds.stream()
            .flatMap(id -> repoProductos.buscarPorId(id).stream())
            .map(p -> new ItemPedido(p, 1)).collect(Collectors.toList());

        if (!validador.validarStock(items)) return Optional.empty();

        double total = calculadora.calcularTotal(items);
        Pedido pedido = new Pedido(0, cliente.get(), items, total);
        int pedidoId = repoPedidos.guardar(pedido);

        notificaciones.enviarConfirmacion(cliente.get(), pedido);
        return Optional.of(pedido);
    }
}
```

## 10. Caso Práctico: Beneficios de la Refactorización

| Aspecto | Antes | Después |
|---|---|---|
| **Cohesión** | Una clase hace todo | Cada clase tiene una responsabilidad |
| **Acoplamiento** | Depende de clases concretas | Depende de interfaces |
| **Testing** | Imposible aislar | Cada componente testeable independientemente |
| **Cambios** | Modificar BD → cambiar múltiples clases | Cambiar BD → solo cambiar implementación del repositorio |
| **Extensión** | Requiere modificar código existente | Agregar nueva funcionalidad = nueva clase |

## 11. Conclusiones

**Cohesión**: Qué tan relacionados están los elementos dentro de un módulo
- **Objetivo**: Alta cohesión funcional (nivel 7)
- **Técnica**: Una responsabilidad por clase (SRP)
- **Beneficio**: Código claro y fácil de mantener

**Acoplamiento**: Qué tan interdependientes son los módulos
- **Objetivo**: Acoplamiento de mensaje (nivel 7)
- **Técnica**: Depender de abstracciones (DIP)
- **Beneficio**: Código flexible y testeable

> **Balance ideal**: Alta cohesión + Bajo acoplamiento = **Código excelente**

---

## 11.1. Checklist de Diseño

**Cohesión**:
- [ ] ¿La clase tiene un nombre descriptivo sin "Y"?
- [ ] ¿Todos los métodos están relacionados con el propósito de la clase?
- [ ] ¿Hay una razón única para que la clase cambie?
- [ ] ¿Los métodos comparten los mismos atributos?

**Acoplamiento**:
- [ ] ¿Las dependencias son interfaces, no clases concretas?
- [ ] ¿Las dependencias se inyectan, no se crean con `new`?
- [ ] ¿Se pasan solo los datos necesarios (no objetos completos)?
- [ ] ¿Se puede testear la clase sin instanciar medio proyecto?
