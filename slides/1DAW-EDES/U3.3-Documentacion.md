# ¿Documentar = comentar código?

---

# Documentar vs comentar

| Accion | Para que sirve | Ejemplo |
|---|---|---|
| Comentar codigo | Aclara una linea o bloque concreto | // Se reintenta por timeout del proveedor |
| Documentar codigo | Describe contratos, uso y comportamiento reutilizable | Javadoc de un metodo publico |
| Documentar proyecto | Explica vision general, arquitectura, instalacion o despliegue | README.md con requisitos y pasos |

- Un comentario suele ser local y cercano a una instruccion.
- La documentacion suele tener alcance mayor y puede ser consultada sin leer toda la implementacion.
- Si un comentario solo traduce el codigo, sobra.
- Si falta contexto para usar o mantener una pieza, hace falta documentacion.

---

# Cuando comentar: necesidad vs codigo claro

- **Mala idea**: usar comentarios para "traducir" codigo confuso.
- **Buena idea**: reescribir codigo para que sea legible.
- Los comentarios deben explicar contexto, intencion y decisiones.
- Regla practica: comentar el **por que**, no repetir el **que**.

**Ejemplo mental**  

`if (x > 18)` no necesita "si x es mayor que 18".  
Si existe una regla de negocio, eso si debe explicarse.

---

# Como escribir comentarios valiosos

| Tipo | Ejemplo | Valor |
|---|---|---|
| Mal comentario | `// suma 1 al contador` | Nulo (obvio) |
| Buen comentario | `// Se reintenta 3 veces por inestabilidad de red del proveedor` | Alto (contexto) |
| Mejor aun | Renombrar `c` por `intentosRestantes` y simplificar logica | Maximo |

- **Claros**: lenguaje directo y preciso.
- **Breves**: sin parrafos innecesarios.
- Con sentido: **aportan informacion nueva**.
- Centrados en el **por que** y el **para que**.
- **Coherentes** con el estado actual del codigo.

# Que significa documentar

- **Documentar** es dejar informacion util para entender, usar, mantener o evolucionar el software.
- **No solo describe codigo**: tambien **explica decisiones, requisitos, arquitectura, despliegue y uso**.
- Puede estar en varios sitios: README, docstrings, Javadoc, wiki, manuales o tickets enlazados.
- Su objetivo no es "rellenar texto", sino **reducir dudas y errores futuros**.

**Ejemplo**
- "Este modulo calcula nominas" es poco.
- "Este modulo calcula nominas segun convenio X, redondea a 2 decimales y excluye contratos en practicas" ya documenta reglas reales.

---

# Tipos de documentos en un proyecto

| Tipo de documento | Contenido habitual | Ejemplo habitual |
|---|---|---|
| Documentacion funcional | Que problema resuelve el sistema y reglas de negocio | Historias de usuario, requisitos, casos de uso |
| Documentacion tecnica | Arquitectura, modulos, APIs, base de datos, dependencias | README tecnico, diagramas, ADR, OpenAPI |
| Documentacion de codigo | Clases, funciones, parametros, retornos y excepciones | Javadoc, docstrings, comentarios puntuales |
| Documentacion operativa | Instalacion, despliegue, copias, monitorizacion | Guia de despliegue, runbook, manual DevOps |
| Documentacion de usuario | Como usar la aplicacion | Manual de usuario, FAQ, tutorial |

- No toda vive en el mismo sitio, pero toda debe estar localizada y mantenerse.
- En proyectos pequeños pueden combinarse varios tipos en pocos documentos bien organizados.

---

# Donde debe ir la documentacion

<div class="two-cols">
<div>

## Buena practica
- **Documentacion cercana al codigo fuente.**
    - Se actualiza junto con los cambios.
    - Reduce contradicciones entre implementacion y texto.
    - Facilita revisiones en `pull request`.
    - Ayuda al mantenimiento correctivo y evolutivo.
    - Permite onboarding mas rapido en equipos.
- Comentarios estructurados en clases, metodos y modulos.
- README y guias enlazadas al repositorio.

</div>

<div>


## Mala practica
- Documento externo que nunca se actualiza.
- Wiki o PDF separado sin relacion con el codigo.


</div>
</div>

# Que conviene documentar (I)

| Que documentar | Ejemplo concreto | Como o donde documentarlo |
|---|---|---|
| Decisiones de diseño relevantes | "Se usa JWT y no sesion en servidor para permitir API stateless" | En `README`, wiki tecnica o ADR del proyecto |
| Interfaces publicas y contratos de uso | Endpoint `POST /usuarios` con campos obligatorios y codigos 201/400 | En OpenAPI, README de API o Javadoc/docstring del controlador |
| Parametros, retornos y excepciones | Metodo `retirar(cantidad)` devuelve nuevo saldo y lanza error si no hay fondos | En Javadoc o docstring de la funcion/metodo |
| Precondiciones y postcondiciones | "La lista debe venir ordenada" o "el pedido queda en estado PAGADO" | En documentacion de metodo y, si es regla de negocio, tambien en pruebas o wiki |
| Efectos secundarios importantes | "Al borrar usuario se eliminan sus tokens activos" | En comentario breve junto al codigo y en documentacion de servicio/API |

---

# Que conviene documentar (II)

| Que documentar | Ejemplo concreto | Como o donde documentarlo |
|---|---|---|
| Limitaciones conocidas y casos borde | "No admite ficheros de mas de 10 MB" o "falla si no hay conexion" | En README, manual de usuario o documentacion de API |
| Ejemplos de uso realistas | Ejemplo de llamada HTTP o ejemplo de crear una `CuentaBancaria` | En README, docstring con ejemplo o coleccion Postman |
| Dependencias o requisitos tecnicos | "Requiere Java 21, MySQL 8 y variable `DB_URL`" | En README de instalacion y scripts de entorno |
| TODOs y partes pendientes | "Pendiente paginacion del listado de facturas" | En issues, backlog o `TODO` cercano al codigo si afecta a un punto concreto |
| Comportamientos no obvios por rendimiento o seguridad | "Se cachea 5 minutos para no saturar la API externa" | En comentario tecnico junto al codigo y en documentacion de arquitectura |

---

# Por que documentar

- Mejora la comprension del sistema.
- Facilita el trabajo en equipo.
- Reduce errores durante mantenimiento.
- Sirve como herramienta de diseno.
- Obliga a pensar antes de programar.

> Documentar bien reduce coste futuro.

# Documentación en Java: Javadoc

<div class="two-cols">
<div>

### ¿Qué es?

- **Javadoc** es el estandar de documentacion en Java.
- Extrae comentarios especiales y genera documentacion de API.
- Ayuda a entender contratos de clases y metodos.
- Se integra bien en procesos de build y CI.

</div>
<div>

### ¿Para que sirve?

- Publicar API interna o externa.
- Mejorar mantenimiento y reutilizacion de codigo.
- Guiar a otros desarrolladores sobre uso correcto.
- Registrar cambios, versiones y deprecaciones.
- Estandarizar la forma de documentar en equipos Java.

</div>
</div>

---

# Elementos que puede documentar Javadoc

- Clases y enums
- Interfaces
- Atributos (fields)
- Constructores
- Metodos
- Paquetes (`package-info.java`)

> Conviene priorizar elementos publicos y protegidos.

---

# Sintaxis base de Javadoc

<div class="two-cols">
<div>

```java
/**
 * Descripcion principal.
 * Explica proposito, contexto y comportamiento.
 */
public class Ejemplo {
}
```
</div>

<div>

- Empieza con `/**` y termina con `*/`.
- La primera frase suele ser resumen breve.
- Despues se anaden detalles y etiquetas (`@param`, `@return`, etc.).

</div>
</div>


---

# Etiquetas Javadoc mas habituales 

<div class="two-cols">
<div>

- `@author`: autor o autores.
- `@version`: version del elemento.
- `@since`: desde que version existe.
- `@deprecated`: elemento obsoleto y alternativa.

</div>
<div>

- `@see`: referencias relacionadas.
- `@param`: describe cada parametro.
- `@return`: describe valor devuelto.
- `@throws`: documenta excepciones lanzadas.

</div>
</div>

# Como generar Javadoc

<br>

```bash
javadoc -d doc src/*.java
```
```bash
javadoc -d build/docs -sourcepath src -subpackages com.miempresa
```
<br>

- Genera un conjunto de paginas HTML.
- Archivo de entrada: clases `.java` del proyecto.
- Resultado tipico: `doc/index.html` como punto de entrada.

# Documentacion en Python

- Python usa **docstrings** para documentar codigo.
- Se colocan dentro de modulos, clases y funciones.
- Herramientas como `pydoc` y `Sphinx` los aprovechan.
- Diferencia clave:
  - comentario normal (`#`) explica lineas puntuales
  - docstring describe API y comportamiento reutilizable

---

# Comentario normal vs docstring vs Javadoc

| Tipo | Sintaxis | Uso principal |
|---|---|---|
| Comentario normal | `# ...` o `// ...` | Aclaracion local |
| Docstring (Python) | `""" ... """` | Documentar modulos/clases/funciones |
| Javadoc (Java) | `/** ... */` | Generar documentacion de API |

- Incluir ejemplos cuando el uso no sea obvio.

# Herramientas en Python: Sphinx

# Buenas practicas y errores

<div class="two-cols">
<div>

### Buenas prácticas

- Documentar interfaces publicas y contratos.
- Evitar documentar lo obvio.
- Mantener documentacion sincronizada con cambios.
- Incluir ejemplos de uso utiles.
- Pensar en quien leera el codigo despues.

</div>
<div>

### Errores frecuentes

- Comentarios obsoletos que contradicen el codigo.
- Descripciones vagas ("hace cosas", "metodo util").
- Explicar linea por linea lo evidente.
- No documentar parametros, retornos o excepciones.
- Separar documentacion del codigo sin proceso de actualizacion.

</div>
</div>

> Regla de oro: documenta para tu compañero y para tu "yo del futuro".
