# ¿Qué es una prueba de software?

> Proceso de **evaluar** un sistema para determinar si cumple los requisitos especificados y funciona correctamente.

Dos objetivos principales:

- **Garantizar la calidad** del software
- **Minimizar el riesgo** de errores en producción

<div class="highlight">
⚠️ Las pruebas demuestran la <strong>presencia</strong> de defectos, no su ausencia.
</div>

# ¿Por qué importa testear?

Los errores detectados tarde son **mucho más caros** de corregir.

| Fase de detección | Coste relativo |
|---|---|
| Durante desarrollo | ×1 |
| En pruebas de sistema | ×10 |
| En producción | ×100 |

- Errores de lógica, fallos de seguridad, problemas de rendimiento
- Un software con bugs daña la reputación del desarrollador
- Detectar pronto = **ahorrar tiempo y dinero**

<!--
NOTA DE PRESENTADOR
-------------------
Esta pirámide es el marco de todo el tema. Nosotros trabajamos
SOLO en el nivel de abajo: pruebas unitarias.

Si preguntan por los otros niveles, esta es la explicación sencilla:

# Pruebas unitarias

Aislan **una unidad de código** (método, clase) y comprueban que funciona correctamente, de forma independiente.

**Ventajas:**
- Se ejecutan en milisegundos
- Detectan errores muy pronto
- Sirven como documentación del código
- Permiten refactorizar con confianza

<div class="highlight">
💡 Una prueba unitaria bien escrita te dice exactamente <em>qué falló y dónde</em>.
</div>

El 80% de los bugs suelen estar en el 20% del código.
Si encuentras muchos fallos en un módulo, busca más ahí.

Nº5 — "Paradoja del pesticida" (de Bruce Beizer):
Si siempre riegas las plantas con el mismo pesticida,
los bichos desarrollan resistencia. Con los tests igual:
si siempre pruebas lo mismo, los bugs que no pruebas sobreviven.
Hay que renovar y ampliar los tests con el tiempo.

Nº7 — "Falacia de la ausencia de errores":
Un software sin bugs que no hace lo que el cliente quería
también es un fracaso. Las pruebas deben verificar los
requisitos CORRECTOS, no solo que el código no explota.

# Principios ISTQB (los 7)

1. Las pruebas demuestran **presencia** de defectos, no ausencia
2. Las pruebas **exhaustivas no existen** (salvo casos triviales)
3. Las pruebas tempranas **reducen costes**
4. Los defectos se concentran en **pocos módulos** (Pareto 80/20)
5. **Paradoja del pesticida**: los mismos tests pierden eficacia con el tiempo
6. Las pruebas **dependen del contexto** (NASA ≠ frutería)
7. **Falacia de ausencia de errores**: sin errores no implica que sea útil

NOTA DE PRESENTADOR
-------------------
TDD es el concepto más contraintuitivo del tema.
Los alumnos (y muchos desarrolladores) lo rechazan al principio
porque parece ilógico: ¿cómo escribo el test si no tengo el código?

La clave está en que el test NO prueba la implementación,
prueba el COMPORTAMIENTO esperado.

Analogía matemática que puede funcionar bien:
"En matemáticas, a veces defines la solución que buscas antes de
encontrarla. Por ejemplo: 'quiero encontrar x tal que x² = 9'.
Eso es el test. Luego encuentras que x = 3 (o x = -3). Eso es el código."

Si un alumno pregunta "¿pero el test no fallará siempre al principio?":
Sí, al principio falla, y eso es correcto y esperado. Lo llamamos RED
porque el indicador del test se pone en rojo. El objetivo es pasar a verde
escribiendo el código mínimo necesario para que funcione.
-->

# TDD — Desarrollo Guiado por Pruebas

<div class="highlight">
Escribir el <strong>test primero</strong>, el código después.
</div>

El ciclo tiene **3 pasos**:

```
  🔴 RED      →   escribe un test que FALLA
  🟢 GREEN    →   escribe el código MÍNIMO para que pase
  🔵 REFACTOR →   mejora el código sin cambiar lo que hace
```

Este ciclo se repite continuamente para cada funcionalidad nueva.

# TDD — ¿Para qué sirve?

> TDD no son pruebas. TDD es una técnica de **diseño de software**.

Te obliga a pensar en el **uso** antes que en la implementación:

- Si no puedes escribir el test → **no tienes claro el requisito**
- Si el test es imposible de escribir → el diseño de tu clase es malo

```
"Nunca escribas funcionalidad sin una prueba que falle antes."
                                                  — Kent Beck
```

AAA es la estructura que los alumnos deben aplicar en TODOS sus tests.
Es lo más práctico de esta sesión.

Analogía con el experimento de laboratorio:
- Arrange = preparar el material y montar el experimento
- Act     = ejecutar el experimento (una sola acción por test)
- Assert  = comparar el resultado con la hipótesis

Sobre el código de la diapositiva:
- @Test es una "anotación" de Java: es una etiqueta que le dice a JUnit
  "este método es un test, ejecútalo automáticamente". En la sesión 2
  se explica con más detalle junto con las demás anotaciones.
- assertEquals(8, resultado) compara el valor esperado (8) con el real
  (lo que devolvió sumar). Si son iguales, el test pasa en verde.
  Si no, falla y muestra: "Expected: 8 but was: 13" (por ejemplo).
- El nombre del método sigue la convención: método_condición_resultado.
  Los guiones bajos son solo para legibilidad; Java los permite en nombres.

# El patrón AAA

Estructura estándar de todo test unitario:

```java
@Test
void sumar_dosNumeros_devuelveSuCorrecta() {

    // ARRANGE — prepara los objetos
    Calculadora calc = new Calculadora();

    // ACT — ejecuta la acción
    int resultado = calc.sumar(3, 5);

    // ASSERT — verifica el resultado
    assertEquals(8, resultado);
}
```

**Arrange · Act · Assert** — siempre en ese orden.

<!--
NOTA DE PRESENTADOR
-------------------
Las clases de equivalencia vienen directamente de la teoría matemática
de relaciones de equivalencia, así que esto probablemente te resulte
familiar. La idea es idéntica: agrupar elementos con el mismo comportamiento
bajo la relación "produce el mismo resultado en el sistema" y elegir
un representante del grupo para probar.

En testing: si sumar(3,5) y sumar(100,200) producen el mismo tipo de
comportamiento (ambos son enteros positivos válidos), no tiene sentido
escribir los dos tests. Con uno del grupo es suficiente.

Los grupos típicos son:
- Valores válidos normales (dentro del rango esperado)
- Valores en el límite exacto (el 0, el máximo permitido, el mínimo)
- Valores inválidos (negativos, nulos, fuera de rango)

Si algún alumno pregunta "¿y si hay un bug solo con el valor 73?",
la respuesta honesta es: "Las pruebas exhaustivas no existen
(principio 2 del ISTQB). Cubrimos los casos razonables, no todos los posibles."
-->

# Clases de equivalencia

No necesitamos probar **todos los valores posibles**, sino un representante de cada grupo con comportamiento similar.

**Ejemplo: campo "edad" que acepta 0–120**

| Clase de equivalencia | Valor de prueba |
|---|---|
| Válida (0–120) | 45 |
| Inválida negativa | −1 |
| Inválida superior | 150 |

Con **3 tests** cubrimos todos los casos relevantes.

NOTA DE PRESENTADOR

Los nombres de los tests son algo que los alumnos suelen hacer

# Nombres de tests

Un buen nombre de test describe **qué** se prueba, **cuándo** y **qué se espera**:

```java
// ❌ Mal
@Test void test1() { ... }

// ❌ Mal — no dice qué se espera
@Test void testSumaConValor3y5() { ... }

// ✅ Bien — describe el comportamiento
@Test
void sumar_conDosPositivos_devuelveLaSuma() { ... }

// ✅ Bien — contexto claro
@Test
void puedePedirFiado_clienteSinDeuda_devuelveTrue() { ... }
```

Convención: `método_condición_resultadoEsperado`
