# Refactorización
## Malos olores y mejora continua del código

**Entornos de Desarrollo · 1º DAW**

---

# Objetivos de la sesión

- Identificar **malos olores** en el código
- Comprender qué es la **refactorización**
- Relacionar ambos conceptos
- Ver un **caso práctico** con código
- Entender por qué los **tests** protegen el proceso

---

# Punto de partida

En muchos proyectos, el código:

- **funciona**
- pero cuesta **leerlo**
- cuesta **modificarlo**
- y resulta fácil **romperlo**

> Ahí suelen aparecer los **malos olores**.

---

# ¿Qué son los malos olores?

Los **malos olores** son **señales** de que el código puede estar mal diseñado o mal estructurado.

- No son necesariamente **errores**
- El programa puede seguir **funcionando**
- Pero indican problemas de:
  - **mantenibilidad**
  - **legibilidad**
  - **escalabilidad**
  - **eficiencia**

---

# Idea clave

Un **mal olor** no significa:

- “el programa está roto”

Un **mal olor** sí significa:

- “el diseño puede mejorarse”

> Detectarlos es el primer paso para aplicar **refactorización**.

---

# Origen del concepto

El término **code smell** fue acuñado por **Kent Beck** y popularizado por **Martin Fowler**.

Se usa para describir características del código que sugieren necesidad de mejora.

- Son **síntomas**
- No son el problema final
- Nos alertan de posible **deuda técnica**

---

# ¿Qué es la refactorización?

La **refactorización** es el proceso de **reestructurar** código existente **sin cambiar su comportamiento externo**.

Su propósito es mejorar:

- la **legibilidad**
- la **estructura**
- la **eficiencia**
- la **mantenibilidad**

---

# Relación entre conceptos

- Los **malos olores** indican dónde mirar
- La **refactorización** permite corregir el diseño
- El objetivo no es añadir funciones nuevas
- El objetivo es **mejorar el código existente**

> **Mal olor** = señal  
> **Refactorización** = respuesta técnica


---

# Ejemplos de malos olores I

| **Mal olor** | **Qué significa** | **Qué problemas provoca** |
|---|---|---|
| **Duplicación de código** | Misma lógica repetida en varios puntos | Cambios duplicados, inconsistencias y más mantenimiento |
| **Métodos largos** | Un método hace demasiadas tareas | Difícil de leer, probar y modificar |
| **Clase grande** | Una clase acumula demasiadas responsabilidades | Baja cohesión, más complejidad y peor mantenimiento |
| **Lista larga de parámetros** | Un método necesita demasiados datos de entrada | Llamadas poco claras y más errores al invocarlo |
| **Comentarios excesivos** | El código necesita demasiada explicación externa | Señal de diseño poco claro y comentarios obsoletos |

---

# Ejemplos de malos olores II

| **Mal olor** | **Qué significa** | **Qué problemas provoca** |
|---|---|---|
| **Código muerto** | Código que ya no se usa | Ruido, confusión y dificultad para mantener |
| **Envidia de características** | Una clase usa demasiado los datos de otra | Acoplamiento alto y poca cohesión |
| **Cadena de mensajes** | Acceso encadenado a muchos objetos internos | Fragilidad y exposición de detalles internos |
| **Cambio divergente** | Una clase cambia por razones distintas | Más errores y responsabilidades mal separadas |
| **Cambio de tiro** | Un cambio obliga a tocar muchas clases | Mantenimiento lento, costoso y arriesgado |

# Ejemplo I: duplicación de código

<div class="two-cols">
  <div>

  ### Mal olor

  La misma validación aparece en dos métodos distintos:

  ```java
  public class UsuarioService {

    public void crearUsuario(String email) {
      if (email == null || email.isBlank() || !email.contains("@")) {
        throw new IllegalArgumentException("Email no válido");
      }
      System.out.println("Usuario creado");
    }

    public void actualizarUsuario(String email) {
      if (email == null || email.isBlank() || !email.contains("@")) {
        throw new IllegalArgumentException("Email no válido");
      }
      System.out.println("Usuario actualizado");
    }
  }
  ```

  </div>
  <div>

  ### Código refactorizado

  Extraemos la validación a un único método:

  ```java
  public class UsuarioService {

    private void validarEmail(String email) {
      if (email == null || email.isBlank() || !email.contains("@")) {
        throw new IllegalArgumentException("Email no válido");
      }
    }

    public void crearUsuario(String email) {
      validarEmail(email);
      System.out.println("Usuario creado");
    }

    public void actualizarUsuario(String email) {
      validarEmail(email);
      System.out.println("Usuario actualizado");
    }
  }
  ```
  </div>
</div>

> **Mejora:** una sola regla de validación y menos riesgo de errores.

# Ejemplo II: método largo

<div class="two-cols">
  <div>

### Mal olor

Un método hace demasiadas cosas a la vez:

```java
public void procesarPedido(Pedido pedido) {
  if (pedido == null) {
    throw new IllegalArgumentException("Pedido nulo");
  }
  if (pedido.getLineas().isEmpty()) {
    throw new IllegalStateException("Pedido vacío");
  }
  double total = 0;
  for (LineaPedido linea : pedido.getLineas()) {
    total += linea.getPrecio() * linea.getCantidad();
  }
  if (total > 100) {
    total = total * 0.9;
  }
  System.out.println("Cliente: " + pedido.getCliente().getNombre());
  System.out.println("Total: " + total);
  guardarEnBaseDeDatos(pedido, total);
  enviarCorreoConfirmacion(pedido.getCliente().getEmail(), total);
}
```

</div>
<div>

### Código refactorizado

Separar tareas hace el código más legible:

```java
public void procesarPedido(Pedido pedido) {
  validarPedido(pedido);
  double total = calcularTotal(pedido);
  mostrarResumen(pedido, total);
  finalizarPedido(pedido, total);
}
```

Se distinguen mejor las responsabilidades:

- **validar**
- **calcular**
- **mostrar**
- **persistir y notificar**

</div>
</div>

# Ejemplo III: clase grande

<div class="two-cols">
<div>

### Mal olor

Una sola clase acumula demasiadas responsabilidades:

```java
public class InformeManager {

  public String leerDatos() { return "datos"; }

  public String calcularEstadisticas() { return "estadísticas"; }

  public void generarPdf() { System.out.println("PDF generado"); }

  public void enviarPorCorreo() { System.out.println("Correo enviado"); }

  public void guardarAuditoria() { System.out.println("Auditoría guardada"); }
}
```

> Lee, calcula, genera, envía y registra: **demasiadas tareas**.

</div>
<div>

### Código refactorizado

Dividir responsabilidades en varias clases:

```java
public class InformeService {
  public String leerDatos() { return "datos"; }
  public String calcularEstadisticas() { return "estadísticas"; }
}

public class InformePdfGenerator {
  public void generarPdf() { System.out.println("PDF generado"); }
}

public class InformeMailer {
  public void enviarPorCorreo() { System.out.println("Correo enviado"); }
}
```

> **Mejora:** cada clase tiene un propósito más claro.

# Ejemplo IV: lista larga de parámetros

<div class="two-cols">
<div>

### Mal olor

Demasiados datos en la firma del método:

```java
public class FacturaService {

  public void crearFactura(
      String nombreCliente,
      String direccion,
      String codigoPostal,
      String ciudad,
      String nif,
      String telefono,
      String email,
      double importe,
      double descuento) {
    System.out.println("Factura creada para " + nombreCliente);
  }
}
```

> Suele indicar que conviene agrupar datos en un **objeto propio**.

</div>
<div>

### Código refactorizado

Agrupar datos relacionados mejora la firma:

```java
public class DatosCliente {
  private String nombreCliente;
  private String direccion;
  private String codigoPostal;
  private String ciudad;
  private String nif;
  private String telefono;
  private String email;
}

public void crearFactura(DatosCliente cliente, double importe, double descuento) {
  System.out.println("Factura creada para " + cliente.getNombreCliente());
}
```

> **Mejora:** menos parámetros y mayor cohesión.

</div>
</div>

# Ejemplo V: comentarios excesivos

<div class="two-cols">
<div>

### Mal olor

Si hace falta comentar todo, quizá el código no es claro:

```java
public double calcularPrecioFinal(double precio, double descuento) {
  // Comprobamos si el descuento es mayor que cero
  if (descuento > 0) {
    // Restamos el descuento al precio
    precio = precio - descuento;
  }
  // Devolvemos el precio final
  return precio;
}
```

> Un mejor nombre o una mejor estructura puede reducir comentarios innecesarios.

</div>
<div>

### Código refactorizado

Un código más expresivo necesita menos comentarios:

```java
public double aplicarDescuento(double precio, double descuento) {
  if (descuento <= 0) {
    return precio;
  }
  return precio - descuento;
}
```

> **Mejora:** el nombre del método y la estructura explican la intención.

</div>
</div>

# Ejemplo VI: código muerto

<div class="two-cols">
<div>

### Mal olor

Código que ya no aporta valor:

```java
public class DescuentoService {

  public double calcularDescuento(double importe) {
    return importe * 0.10;
  }

  public double calcularDescuentoAntiguo(double importe) {
    return importe * 0.05;
  }
}
```

Si `calcularDescuentoAntiguo()` ya no se usa:

- añade ruido
- dificulta entender la clase

</div>
<div>

### Código refactorizado

La mejora suele ser directa:

```java
public class DescuentoService {

  public double calcularDescuento(double importe) {
    return importe * 0.10;
  }
}
```

> **Mejora:** menos ruido y una clase más fácil de mantener.

</div>
</div>

# Ejemplo VII: envidia de características

<div class="two-cols">
<div>

### Mal olor

Una clase usa demasiados datos de otra:

```java
public class Cliente {
  private String nombre;
  private String ciudad;

  public Cliente(String nombre, String ciudad) {
    this.nombre = nombre;
    this.ciudad = ciudad;
  }

  public String getNombre() { return nombre; }
  public String getCiudad() { return ciudad; }
}

public class PedidoPrinter {
  public void imprimir(Cliente cliente) {
    System.out.println(cliente.getNombre().toUpperCase());
    System.out.println(cliente.getCiudad().toUpperCase());
  }
}
```

</div>
<div>

### Código refactorizado

Mover comportamiento al objeto que posee los datos:

```java
public class Cliente {
  private String nombre;
  private String ciudad;

  public String getResumen() {
    return nombre.toUpperCase() + " - " + ciudad.toUpperCase();
  }
}

public class PedidoPrinter {
  public void imprimir(Cliente cliente) {
    System.out.println(cliente.getResumen());
  }
}
```

> **Mejora:** más cohesión y menos dependencia externa.

</div>
</div>

# Ejemplo VIII: cadena de mensajes

<div class="two-cols">
<div>

### Mal olor

El código navega por demasiados objetos:

```java
public class PedidoService {
  public String obtenerCiudad(Pedido pedido) {
    return pedido.getCliente()
        .getDireccion()
        .getCiudad()
        .getNombre();
  }
}
```

Problemas:
- alto acoplamiento
- fragilidad ante cambios internos

</div>
<div>

### Código refactorizado

Ocultar la navegación interna detrás de un método:

```java
public class Pedido {
  private Cliente cliente;

  public String getNombreCiudadCliente() {
    return cliente.getDireccion().getCiudad().getNombre();
  }
}

public class PedidoService {
  public String obtenerCiudad(Pedido pedido) {
    return pedido.getNombreCiudadCliente();
  }
}
```

> **Mejora:** el cliente del objeto conoce menos detalles internos.

</div>
</div>

# Malos olores vs Refactorización I

| **Mal olor** | **Síntoma** | **Consecuencia** | **Refactorización sugerida** |
|---|---|---|---|
| **Duplicación de código** | Misma lógica repetida | Inconsistencias y cambios repetidos | **Extraer método** |
| **Método largo** | Un bloque hace demasiadas tareas | Baja legibilidad y pruebas difíciles | **Dividir en métodos** |
| **Clase grande** | Muchas responsabilidades en una clase | Complejidad y baja cohesión | **Separar responsabilidades** |
| **Lista larga de parámetros** | Firmas con demasiados datos | Llamadas confusas y errores | **Introducir objeto parámetro** |
| **Comentarios excesivos** | El código necesita explicación continua | Diseño poco expresivo | **Renombrar y simplificar** |

# Malos olores vs Refactorización II

| **Mal olor** | **Síntoma** | **Consecuencia** | **Refactorización sugerida** |
|---|---|---|---|
| **Código muerto** | Métodos o variables sin uso | Ruido y confusión | **Eliminar código muerto** |
| **Envidia de características** | Una clase usa mucho otra | Acoplamiento alto | **Mover método** |
| **Cadena de mensajes** | Demasiados accesos encadenados | Fragilidad ante cambios | **Ocultar delegación** |
| **Cambio divergente** | Una clase cambia por varios motivos | Más errores y mezcla de roles | **Dividir clase** |
| **Cambio de tiro** | Un cambio obliga a tocar muchas clases | Mantenimiento lento y costoso | **Reducir acoplamiento** |

# Pruebas antes y después

Para refactorizar con seguridad hacen falta **tests**.

Los tests permiten:

- comprobar que el comportamiento sigue igual
- detectar errores introducidos durante el cambio
- refactorizar con mayor confianza

> **Sin pruebas, refactorizar es mucho más arriesgado**.

---


# Calidad automática del código

Además de los tests, podemos apoyar la calidad con **análisis estático**.

Este análisis revisa el código **sin ejecutarlo** para detectar:

- errores potenciales
- vulnerabilidades
- problemas de diseño
- incumplimientos de estilo

---

# ¿Qué aporta el análisis estático?

- Es más **rápido** que una revisión manual completa
- Detecta problemas que pueden pasar desapercibidos
- Ayuda a mantener convenciones comunes de equipo
- Mejora la **mantenibilidad** del proyecto
- Refuerza la calidad antes de desplegar

---

# Analizadores populares (Java/Kotlin/Android)

- **Ktlint**: estilo y formato en Kotlin
- **Detekt**: reglas de calidad y *code smells* en Kotlin
- **FindBugs**: análisis de bytecode Java (`.class`)
- **Checkstyle**: convenciones y estilo de código Java
- **Android Lint**: chequeos de calidad en proyectos Android

# Caso práctico del documento

### Mal olor: **Duplicación de código**

Tenemos una clase que calcula salario:

- un método para salario **mensual**
- otro método para salario **semanal**

Ambos repiten exactamente la misma lógica.

---

# Qué vamos a observar

En el código original veremos:

- cálculo del **salario base**
- cálculo de la **bonificación**
- misma lógica en dos métodos distintos

Eso provoca:

- más trabajo de mantenimiento
- más riesgo de inconsistencias
- peor evolución futura

---

# Código original con duplicación

```java
public class CalculadoraSalario {

    public double calcularSalarioMensual(int horasTrabajadas, double tarifaPorHora) {
        double salarioBase = horasTrabajadas * tarifaPorHora;
        double bonificacion = salarioBase * 0.10;
        return salarioBase + bonificacion;
    }

    public double calcularSalarioSemanal(int horasTrabajadas, double tarifaPorHora) {
        double salarioBase = horasTrabajadas * tarifaPorHora;
        double bonificacion = salarioBase * 0.10;
        return salarioBase + bonificacion;
    }
}
```

---

# ¿Por qué es un mal olor?

Porque la lógica de cálculo está **duplicada** en:

- `calcularSalarioMensual()`
- `calcularSalarioSemanal()`

Si mañana cambia la fórmula:

- hay que modificar **varios lugares**
- puede olvidarse uno
- aparecen **errores e inconsistencias**

---

# Refactorización aplicada

La mejora consiste en:

- **extraer** la lógica común
- crear un método reutilizable
- hacer que ambos métodos deleguen en él

Resultado:

- menos duplicación
- menos riesgo
- más claridad

---

# Código refactorizado en Java

```java
public class CalculadoraSalario {

    private double calcularSalarioTotal(int horasTrabajadas, double tarifaPorHora) {
        double salarioBase = horasTrabajadas * tarifaPorHora;
        double bonificacion = salarioBase * 0.10;
        return salarioBase + bonificacion;
    }

    public double calcularSalarioMensual(int horasTrabajadas, double tarifaPorHora) {
        return calcularSalarioTotal(horasTrabajadas, tarifaPorHora);
    }

    public double calcularSalarioSemanal(int horasTrabajadas, double tarifaPorHora) {
        return calcularSalarioTotal(horasTrabajadas, tarifaPorHora);
    }
}
```

---

# Resultado

<div class="two-cols">
<div>

### ¿Que se ha extraido?

Se ha extraído un método privado:

- `calcularSalarioTotal(int horasTrabajadas, double tarifaPorHora)`

Ese método centraliza:

- el cálculo del **salario base**
- el cálculo de la **bonificación**
- el **resultado final**

</div>
<div>

### Mejora conseguida

Con la refactorización:

- desaparece la **duplicación**
- el código queda más **limpio**
- la fórmula se mantiene en **un único punto**
- los cambios futuros son más **seguros**

</div>
</div>

> Misma funcionalidad, mejor diseño.

---

# Recuerda

> Refactorizar no es “reescribir por gusto”.

Refactorizar es:

- mejorar estructura
- reducir deuda técnica
- facilitar mantenimiento
- apoyar la evolución del software

Y siempre:

- **sin cambiar el comportamiento externo**

---

# Conclusión

Los **malos olores** ayudan a detectar problemas de diseño.

La **refactorización** permite corregirlos de forma controlada.

Los **tests** aseguran que la mejora interna no rompa el sistema.

> La calidad del código se trabaja de forma **continua**.

---

# Resumen final

- **Malos olores** = señales de mal diseño
- **Refactorización** = mejora interna sin cambiar funcionalidad
- **Duplicación** = un olor clásico
- **Tests** = protección imprescindible
